## 1. 架构概述

### 1.1 架构风格

MaaS平台采用**微服务架构**，遵循以下设计哲学：

- **单一职责**：每个服务只负责一个明确的业务域
- **API优先**：服务间通过明确定义的接口通信，不直接共享数据库
- **事件驱动**：高吞吐量的日志、计费、监控数据通过消息队列异步处理
- **无状态服务**：核心请求处理路径上的服务均为无状态，支持水平扩展
- **可观测优先**：每个服务内建Metrics、Tracing、Logging支持

## 3. API网关详细设计

### 3.1 请求处理管道（Pipeline）

```
HTTP请求到达
    │
    ▼
[1] SSL卸载（Nginx/Envoy）
    │
    ▼
[2] 认证中间件
    ├─ 提取 Authorization Header
    ├─ 查询 Redis 缓存（Key状态+元数据）
    ├─ 若缓存未命中，调用 auth-service gRPC
    └─ 注入 RequestContext（tenant_id, key_id, quota...）
    │
    ▼
[3] 限流中间件（令牌桶算法）
    ├─ 检查 RPM 桶（Redis INCR + EXPIRE）
    ├─ 检查 TPM 桶（预估Token数）
    └─ 检查 IP 黑名单
    │
    ▼
[4] 预算检查中间件
    ├─ 查询当前 api_key 剩余额度
    └─ 预估本次请求 Token 消耗
    │
    ▼
[5] 请求标准化中间件
    ├─ 提取 model 字段
    └─ 请求体验证（JSON Schema）
    │
    ▼
[6] 路由分发（调用 routing-service）
    │
    ▼
[7] 响应标准化中间件
    ├─ 注入 x_maas 扩展字段
    └─ 统一错误码映射
    │
    ▼
[8] 日志埋点中间件（异步）
    └─ 发送事件至 Kafka
```

### 3.2 API Key 缓存设计

为避免每次请求都查询数据库，API Key 信息缓存在 Redis 中：

```
Key:   auth:apikey:{key_hash}
Value: JSON {
  "id": "uuid",
  "tenant_id": "uuid",
  "project_id": "uuid|null",
  "status": "active",
  "token_quota": 10000000,
  "token_used": 5000000,
  "rate_limit_rpm": 600,
  "ip_whitelist": ["192.168.1.0/24"],
  "expires_at": "2027-01-01T00:00:00Z"
}
TTL:   60秒
```

**缓存失效策略：**
- TTL到期自动失效（60秒内的状态变更有延迟）
- 吊销/禁用Key时，主动调用 `DEL auth:apikey:{key_hash}` 即时失效

### 3.3 流式响应（Streaming）处理

```go
// 伪代码：流式响应代理
func HandleStreamingRequest(w http.ResponseWriter, r *http.Request) {
    // 1. 认证+路由（同普通请求）
    endpoint := routingService.Select(ctx, routingKey)
    
    // 2. 向上游发起流式请求
    upstreamResp, err := adapterService.StreamChat(ctx, endpoint, req)
    
    // 3. 设置响应头
    w.Header().Set("Content-Type", "text/event-stream")
    w.Header().Set("Cache-Control", "no-cache")
    w.Header().Set("X-Accel-Buffering", "no") // 禁用Nginx缓冲
    
    flusher := w.(http.Flusher)
    totalTokens := 0
    
    // 4. 逐chunk转发
    for chunk := range upstreamResp.Chunks() {
        data := standardizeChunk(chunk)
        fmt.Fprintf(w, "data: %s\n\n", data)
        flusher.Flush()
        totalTokens += chunk.Tokens
    }
    
    // 5. 发送结束标记
    fmt.Fprintf(w, "data: [DONE]\n\n")
    flusher.Flush()
    
    // 6. 异步记录日志
    go logUsage(ctx, totalTokens)
}
```

## 4. 智能路由引擎详细设计

### 4.1 路由决策算法实现

```python
# 伪代码：路由评分算法
def select_endpoint(request_ctx, candidate_endpoints, policy):
    """
    多维加权评分，选择最优模型端点
    """
    weights = policy.weight_config  # 从路由策略配置中读取
    scores = []
    
    for ep in candidate_endpoints:
        if ep.health_status != "healthy":
            continue
        
        # 1. 成本评分（越低越好，取倒数归一化）
        estimated_cost = estimate_cost(ep, request_ctx.estimated_tokens)
        cost_score = 1.0 / (estimated_cost + 0.0001)  # 防止除零
        
        # 2. 延迟评分（EWMA P95，越低越好，取倒数归一化）
        latency_score = 1.0 / (ep.last_latency_p95_ms + 1)
        
        # 3. 能力匹配度评分（0-1，基于请求特征与模型标签匹配）
        match_score = calculate_capability_match(request_ctx, ep.model)
        
        # 4. 供应商稳定性评分（近1小时成功率）
        stability_score = ep.success_rate_1h
        
        # 加权求和
        total_score = (
            weights.cost_weight * normalize(cost_score) +
            weights.latency_weight * normalize(latency_score) +
            weights.match_weight * match_score +
            weights.stability_weight * stability_score
        )
        
        scores.append((ep, total_score))
    
    if not scores:
        raise ModelUnavailableError("没有可用的模型端点")
    
    # 选择得分最高的端点
    return max(scores, key=lambda x: x[1])[0]
```

### 4.2 端点健康状态管理

```
健康检查服务（每30秒执行）：
    ├─ 向每个端点发送 GET /health 或发送轻量测试请求
    ├─ 记录响应时间和状态码
    ├─ 连续3次失败 → 标记为 "unhealthy"
    ├─ 1次成功 + 连续5次成功 → 标记为 "healthy"
    └─ 更新 Redis：routing:endpoint:{id}:health

降级策略：
    ├─ unhealthy 端点：从候选池中移除，不参与路由
    ├─ degraded 端点：保留但权重降低50%
    └─ 定期（5分钟）重试 unhealthy 端点，检测是否恢复
```

### 4.3 故障转移流程

```
调用失败（超时/5xx/限流）
    │
    ▼
检查重试配置（max_retries=3, backoff=exponential）
    │
    ▼
选择备用端点（优先从 fallback_models 列表选择）
    │
    ├─ 备用端点存在 → 路由至备用端点，记录故障转移日志
    │
    └─ 无可用备用端点 → 返回 503 model_unavailable
    
故障记录：
    ├─ 记录失败端点ID、失败原因、时间戳至 Kafka
    └─ 连续失败计数达到阈值 → 触发告警通知
```

## 6. 计量计费系统详细设计

### 6.1 Token计量精度保障

**计量逻辑：**
1. 优先使用**模型返回的usage字段**（最准确）
2. 若模型未返回usage（部分国产模型），使用**tiktoken本地计算**
3. 流式响应中，实时累计各chunk的token数，最终用usage字段修正

**幂等保障：**
```
每次API调用生成唯一 request_id
计费服务处理前先检查：
    SELECT id FROM billing.usage_records WHERE request_id = $1
如已存在则跳过（避免Kafka消费重试导致重复计费）
```

### 6.2 Kafka Topic设计

| Topic名称 | 分区数 | 副本数 | 用途 |
|---------|--------|--------|------|
| `maas.api.requests` | 12 | 3 | API请求完成事件（计费用） |
| `maas.routing.decisions` | 6 | 3 | 路由决策事件（策略分析用） |
| `maas.model.events` | 3 | 3 | 模型状态变更事件 |
| `maas.alerts` | 3 | 3 | 告警通知事件 |

**`maas.api.requests` 消息格式：**
```json
{
  "schema_version": "1.0",
  "event_type": "api_request_completed",
  "timestamp": "2026-05-14T08:00:00.123Z",
  "request_id": "req_abcdef123456",
  "tenant_id": "uuid",
  "project_id": "uuid|null",
  "api_key_id": "uuid",
  "model_id": "uuid",
  "endpoint_id": "uuid",
  "prompt_tokens": 150,
  "completion_tokens": 320,
  "total_tokens": 470,
  "latency_ms": 342,
  "status_code": 200,
  "is_cached": false,
  "routing_policy_id": "uuid|null"
}
```

### 6.3 实时额度更新

```
请求完成 → Kafka 事件 → 计费消费者
    │
    ▼
1. 写入 usage_records 明细（PostgreSQL）
2. 原子更新 Redis 累计值：
   INCRBY billing:usage:{api_key_id}:tokens:total {total_tokens}
   INCRBY billing:usage:{api_key_id}:cost:total {cost_microcents}
3. 检查是否触发预算告警（与 budget_configs 中阈值比较）
4. 若触发：发送事件至 maas.alerts topic
```

## 7. 异步消息流设计

### 7.1 完整消息流拓扑

```
API Gateway
    │ 发布（同步请求完成后，异步发布）
    ▼
Kafka: maas.api.requests
    │
    ├──► Consumer Group: billing-consumers（计量计费）
    │         └── 写入 PostgreSQL usage_records
    │         └── 更新 Redis 实时额度
    │
    ├──► Consumer Group: metrics-consumers（监控指标）
    │         └── 聚合写入 VictoriaMetrics
    │
    └──► Consumer Group: audit-consumers（审计日志）
              └── 写入 Elasticsearch

Routing Engine
    │ 发布（每次路由决策后）
    ▼
Kafka: maas.routing.decisions
    └──► Consumer Group: analytics-consumers（路由效果分析）
              └── 存入数仓/Elasticsearch

Billing Service
    │ 发布（预算告警触发时）
    ▼
Kafka: maas.alerts
    └──► Consumer Group: notification-consumers（通知服务）
              ├── 发送钉钉 Webhook
              └── 发送邮件（SMTP）
```

### 7.2 消费者可靠性保障

- **手动提交Offset**：处理完成后才提交，避免数据丢失
- **死信队列（DLQ）**：处理失败超过3次的消息发送至 `maas.*.dlq` Topic，人工排查
- **消费幂等**：通过 `request_id` 唯一键保障重复消费时不产生重复写入

## 8. 缓存策略设计

### 8.1 Redis 缓存命名规范

```
auth:apikey:{key_hash}              # API Key信息缓存，TTL=60s
auth:ratelimit:{key_id}:rpm         # RPM限流计数器，TTL=60s
auth:ratelimit:{key_id}:tpm         # TPM限流计数器，TTL=60s
billing:usage:{key_id}:tokens       # 当月Token累计（用于额度检查）
routing:endpoint:{id}:health        # 端点健康状态，TTL=45s
routing:endpoint:{id}:metrics       # 端点实时指标（延迟、负载），TTL=30s
cache:l1:{prompt_md5}               # 语义缓存L1（精确匹配），TTL=24h
```

### 8.2 缓存容量规划

| 缓存Key类型 | 单条大小 | 预估条数 | 总容量 |
|---------|--------|--------|--------|
| API Key信息 | ~500B | 10,000 | ~5MB |
| 限流计数器 | ~20B | 100,000 | ~2MB |
| 端点健康/指标 | ~200B | 1,000 | ~200KB |
| 语义缓存L1 | ~2KB | 500,000 | ~1GB |
| 账单实时累计 | ~50B | 50,000 | ~2.5MB |
| **合计（+ 预留）** | - | - | **~4GB** |

**Redis集群规格建议：** 3主3从，每节点8GB内存，预留50%空间。

|------|---------|------|
