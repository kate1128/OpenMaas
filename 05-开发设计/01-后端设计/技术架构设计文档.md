## 1. 架构概述

### 1.1 架构风格

MaaS平台采用**微服务架构**，遵循以下设计哲学：

- **单一职责**：每个服务只负责一个明确的业务域
- **API优先**：服务间通过明确定义的接口通信，不直接共享数据库
- **事件驱动**：高吞吐量的日志、计费、监控数据通过消息队列异步处理
- **无状态服务**：核心请求处理路径上的服务均为无状态，支持水平扩展
- **可观测优先**：每个服务内建Metrics、Tracing、Logging支持

## 2. 微服务拆分方案

### 2.1 服务列表

```

├── gateway-service              # 统一API网关（多协议接入：OpenAI/Anthropic/Gemini/WS）
```

## 3. API网关详细设计

### 3.1 请求处理管道（Pipeline — 11步中间件链）

```
HTTP请求到达
    │
    ▼
[0] SSL卸载（Nginx/Envoy）+ TraceID/RequestID 注入
    │
    ▼
[1] 协议检测中间件（V2.0 新增）
    ├─ URL 前缀判断：/v1/chat/completions → OpenAI / /v1/anthropic/ → Anthropic / /v1/gemini/ → Gemini
    ├─ 注入 protocol_origin → request.Context
    └─ 后续使用统一 StandardRequest，响应时反向转换
    │
    ▼
[2] 认证中间件（双模：API Key + SSO JWT）
    ├─ 提取 Authorization: Bearer {api_key} 或 X-MaaS-Token: {jwt}
    ├─ Redis GET auth:apikey:{key_hash}，未命中 → gRPC auth-service.ValidateApiKey / ValidateJWT
    ├─ 校验 status(active)、IP白名单、过期时间、scope
    └─ 注入 ApiKeyContext / UserContext → request.Context
    │
    ▼
[3] 限流中间件（令牌桶算法）
    ├─ RPM桶（Redis INCR + EXPIRE 60s）
    ├─ TPM桶（预估 Token 数）
    └─ Key级 / 项目级 / 租户级三档限流
    │
    ▼
[4] 预算预检中间件
    ├─ gRPC billing-service.PreCheckQuota(tenant_id, est_tokens)
    └─ 超额前置拒绝（ALLOW / DENY）
    │
    ▼
[5] 合规前置过滤中间件（V2.0 新增）
    ├─ gRPC compliance-service.CheckDataResidency(tenant_id, model_id, client_region)
    ├─ ALLOW / DENY → 451 Unavailable For Legal Reasons / REDIRECT（注入 X-MaaS-Required-Region）
    └─ 注入 zero_retention / kms_key_id 标记
    │
    ▼
[6] 内容安全预检中间件（V2.0 新增）
    ├─ 本地轻量关键词过滤（P99 < 2ms）
    └─ 严重违规直接拦截（403 content_policy_violation）
    │
    ▼
[7] 请求标准化中间件
    ├─ 校验 model 字段为有效逻辑模型名（正则 ^maas:.+）
    ├─ 补全缺省字段（stream: false 等）
    └─ 注入 zero_retention / kms_key_id 标记
    │
    ▼
[8] 路由分发（gRPC routing-service.RouteRequest → RouteDecision{backend_id, fallback_chain}）
    │
    ▼
[9] 供应商调用
    ├─ 同步请求：gRPC adapter-service.CallVendorBackend
    └─ 流式请求：HTTP POST adapter-service /v1/chat/completions/stream（避免 SSE 穿越两层 gRPC）
    │
    ▼
[10] 响应标准化中间件
    ├─ 根据 protocol_origin 反向转换为客户端期望格式
    └─ 注入 x_maas_request_id / x_maas_trace_id / x_maas_tenant_id
    │
    ▼
[11] 事件埋点中间件（异步）
    └─ 发送 api_request_completed 事件至 Kafka maas.api.requests
```

### 3.2 API Key 缓存设计（PRD V2.0 5种Key类型）

为避免每次请求都查询数据库，API Key 信息缓存在 Redis 中：

```
Key:   auth:apikey:{key_hash}
Value: JSON {
  "id": "uuid",
  "tenant_id": "uuid",
  "project_id": "uuid|null",
  "status": "active",
  "key_type": "mk-prod",          // V2.0新增：mk-prod / mk-dev / mk-ci / mk-pg / mk-sub
  "token_quota": 10000000,
  "token_used": 5000000,
  "rate_limit_rpm": 600,
  "ip_whitelist": ["192.168.1.0/24"],
  "scope": ["chat:completion"],   // V2.0新增：权限范围
  "zdr_mode": "DISABLED",         // V2.0新增：零数据保留模式
  "expires_at": "2027-01-01T00:00:00Z"
}
TTL:   60秒
```

**缓存失效策略：**
- TTL到期自动失效（60秒内的状态变更有延迟）
- 吊销/禁用Key时，主动调用 `DEL auth:apikey:{key_hash}` 即时失效
- Key 泄露检测（异常调用模式）→ 自动 SUSPENDED + 告警

### 3.3 流式响应（Streaming）处理（V2.0 修订）

> **架构决策**：流式请求不再通过 routing-service gRPC 代理。gateway 从 routing 获取 RouteDecision 后，**直接通过 HTTP 调用 adapter-service**，避免 SSE 流穿越两层 gRPC 导致的背压失效。

```
流式请求链路（V2.0）：
  Dev → gateway-service (HTTP SSE)
        │
        ├── gRPC RouteRequest → routing-service → RouteDecision{backend_id, fallback_chain}
        │
        └── HTTP POST /v1/chat/completions/stream → adapter-service
                                                        │
                                                        ├── KeyPool.get_key(backend_id)
                                                        ├── litellm.acompletion(stream=True, ...)
                                                        └── SSEStreamWrapper → HTTP SSE 透传
背压通过 HTTP → HTTP 自然传播
```

## 4. 智能路由引擎详细设计

### 4.1 路由决策算法实现（PRD V2.0 四维评分）

```
四维评分公式（PRD §03 4.1）：
  Score_i = W₁ × cost_score_i + W₂ × latency_score_i + W₃ × capability_match_i + W₄ × health_score_i

其中：
  cost_score_i  = 1 / (estimated_cost_i + ε)              — 成本越低分越高
  latency_score_i = 1 / (EWMA_P95_latency_i + 1)           — 延迟越低分越高
  capability_match_i = feature intersection / required      — 能力标签匹配率
  health_score_i = 0.30×success_rate + 0.20×latency_norm + 0.20×throttle_norm + 0.20×quota_norm + 0.10×incident_norm
```

### 4.2 策略类型（PRD V2.0 五种策略）

| 策略类型 | 适用场景 | 说明 |
|---------|---------|------|
| **COST_OPTIMAL** | 批量处理、非实时任务 | 最低成本优先 |
| **PERFORMANCE** | 实时交互、低延迟要求 | 最低延迟优先 |
| **FIXED_MODEL** | 指定模型固定调用 | 直接路由至指定 backend_id |
| **WEIGHTED_ROUND_ROBIN** | 同质实例间均衡 | 加权轮询 |
| **CANARY** | A/B 实验 / 灰度发布 | 按流量比例分割（prompt-eval-service 联动） |

### 4.3 Fallback 链（L1→L5 降级）

```
L1: 主后端（四维评分最高 candidate）
L2: 同供应商其他后端（同 vendor 其他 backend）
L3: 跨供应商同档模型（capability_match ≥ 0.8）
L4: 跨供应商降级模型（capability_match ≥ 0.6）
L5: 缓存兜底（语义缓存命中或静态 fallback 响应）
```

### 4.4 端点健康状态管理

```
健康检查（每30秒）：
  ├─ 向每个 vendor_backend 发送 /health 探活
  ├─ 连续3次失败 → 标记为 "unhealthy"，触发健康评分更新事件
  ├─ 连续5次成功 → 标记为 "healthy"
  └─ 5维健康评分实时更新（success_rate / latency / throttle / quota / incident）

降级策略：
  ├─ unhealthy 端点：从候选池移除
  ├─ degraded 端点：权重降低 50%
  └─ 熔断器 OPEN：adapter-service 侧自动拦截（不进入候选池）
```

### 4.5 A/B 实验联动（PRD V2.0）

prompt-eval-service 创建 CANARY 策略 → routing-service 按流量比例分流 → 统计显著性检验 → 实验结论触发自动 promote / rollback。

### 4.6 质量门禁联动（PRD V2.0）

eval 结果 → quality_score 更新 → Kafka maas.model.events → routing-service 自动调整权重（5%-15% 阈值）。

## 5. 模型适配器框架设计（V2.0 纯Python架构）

### 5.1 架构决策

adapter-service 是**唯一的纯 Python 核心服务**（Go Sidecar 方案已弃用）。直接 import LiteLLM，零 IPC 开销。LiteLLM 定位为"协议翻译器"，不做负载均衡和重试（routing-service 已做后端选择，自研 KeyPool / RetryEngine / CircuitBreaker）。

详见 `微服务设计/04-adapter-service详细设计.md` §0。

### 5.2 核心调用链路

```
gateway/routing → gRPC CallVendorBackend(backend_id, openai_body, stream, ...)
    │
    ▼
adapter-service (Python FastAPI)
    ├── KeyPool.get_key(backend_id)      # 从 Redis Key 池选择可用 Key
    ├── CircuitBreaker.is_allowed(backend_id)  # 熔断检查
    ├── litellm.acompletion(             # LiteLLM 协议翻译 + HTTP 调用
    │       model="openai/gpt-4o",       # LiteLLM 通过 model 前缀识别 100+ 供应商协议
    │       messages=openai_body,
    │       api_key=selected_key,
    │       stream=True/False,
    │       extra_body={...},            # Prompt Cache cache_control 透传
    │   )
    ├── RetryEngine (同一 backend 指数退避，max=3)
    └── 语义缓存（Embedding 相似度 ≥ 0.92，Qdrant/Redis Vector）
```

### 5.3 协议适配

LiteLLM 根据 `model` 前缀自动识别供应商协议并转换：
- `openai/` → OpenAI 协议
- `anthropic/` → Anthropic 协议（含 cache_control 透传）
- `gemini/` → Gemini 协议（自动缓存，识别 cachedContentTokenCount）
- `azure/` → Azure OpenAI
- `dashscope/` → 阿里云 DashScope
- `deepseek/` → DeepSeek（OpenAI 兼容）

无需手工维护各供应商的参数映射，LiteLLM 内置 100+ 供应商协议处理。

## 6. 计量计费系统详细设计

### 6.1 Token计量精度保障（三优先级 — PRD §06 1.2）

1. **供应商 usage 字段**（最高优先级）：OpenAI/Anthropic 等返回 usage.prompt_tokens/completion_tokens，以此为准
2. **网关侧 Tokenizer 计算**：供应商未返回 usage 时，使用对应模型 Tokenizer（tiktoken/sentencepiece）实时计算
3. **估算值**（字符数/4）：Tokenizer 超时或不可用时兜底，`is_estimated = true`

**多模态 Token 折算**（PRD §06 1.2.2）：图片 85 Token/张（低分辨率）、170 Token/512px-tile（高分辨率），音频 1 Token/0.1秒。

**Cached Token 折扣**（PRD §06 1.2.3）：KV Cache 命中的 Token 以 `cached_input_tokens` 独立记录，以标准 Prompt Token 价格的 10% 计费。

### 6.2 预扣-核销两阶段（PRD §06 1.4）

```
请求前（PreCheckQuota）：
  gateway → gRPC billing-service.PreCheckQuota(tenant_id, est_tokens)
  → 检查四层预算（平台→租户→项目→Key）
  → ALLOW / DENY(quota_exceeded)

请求后（核销）：
  Kafka api_request_completed → billing consumer
  → 幂等检查（request_id 去重）
  → INSERT INTO billing_ledger (Append-Only)
  → Redis DECR billing:balance:{tenant_id}
  → 检查预算阈值（50%/80%/100%）→ 发布告警
```

### 6.3 四层预算层级（PRD §06 第2章）

```
平台级预算 → 租户级预算 → 项目级预算 → Key级预算
  硬限：超额立即拒绝      软限：超额允许但触发告警
```

合同定价模型（PRD §06 第7章）：FLAT_RATE / TIERED_VOLUME / TIERED_COMMITMENT / PACKAGE。

### 6.4 成本异常检测（PRD §06 第4章）

三种检测算法：
- **算法 A：Z-Score**（实时）— 窗口 7天×24小时，Z > 3.0 触发 SPIKE 告警
- **算法 B：Prophet 时间序列分解**（每日批处理）— 趋势漂移检测，需要 ≥ 30天历史
- **算法 C：KL 散度**（每日批处理）— 模型分布异常检测，KL > 0.5 告警

五种异常类型：SPIKE / TREND_DRIFT / MODEL_DIST / INEFFICIENCY / BUDGET_PENETRATION。

### 6.5 Kafka Topic设计

| Topic名称 | 分区数 | 副本数 | 用途 |

## 7. 异步消息流设计

### 7.1 完整消息流拓扑（V2.0）

```
API Gateway
    │ 发布（请求完成后异步）
    ▼
Kafka: maas.api.requests
    │
    ├──► Consumer Group: billing-consumers（计量计费）
    │         └── 写入 billing_ledger（Append-Only）
    │         └── 更新 Redis 实时额度
    │
    ├──► Consumer Group: trace-consumers（LLMOps Trace）
    │         └── 写入 ClickHouse trace 表（44字段）
    │
    └──► Consumer Group: compliance-consumers（合规审计）
              └── 写入 PostgreSQL audit_log（哈希链）

Routing Engine
    │ 发布（每次路由决策后）
    ▼
Kafka: maas.routing.decisions
    └──► Consumer Group: analytics-consumers（路由效果分析）
              └── 策略效果回测 + 模拟引擎数据源

Billing Service
    │ 发布（预算告警/异常检测触发时）
    ▼
Kafka: maas.billing.alerts / maas.anomaly.alerts
    └──► Consumer Group: notification-consumers
              ├── 钉钉/企业微信 Webhook
              ├── 邮件（SMTP）
              └── PagerDuty

Compliance Service
    │ 发布（内容安全违规/注入检测）
    ▼
Kafka: maas.compliance.events
    └──► Consumer Group: notification + audit

Model Catalog Service
    │ 发布（生命周期变更/价格更新/健康变化）
    ▼
Kafka: maas.model.events
    └──► Consumer Group: adapter（语义缓存失效）+ routing（策略刷新）
```

### 7.2 消费者可靠性保障

- **手动提交Offset**：处理完成后才提交，避免数据丢失
- **死信队列（DLQ）**：处理失败超过3次的消息发送至 `maas.*.dlq` Topic，人工排查
- **消费幂等**：通过 `request_id` 唯一键保障重复消费时不产生重复写入
- **有序处理**：同一 `request_id` 的事件由同一分区保证顺序

## 11. 多协议支持策略（PRD V2.0 §00 架构）

### 11.1 端点矩阵

| 协议 | 端点 | 说明 |
|------|------|------|
| **OpenAI** | `POST /v1/chat/completions` | 主对话接口（HTTP/SSE） |
| | `POST /v1/completions` | 文本补全 |
| | `POST /v1/embeddings` | Embedding |
| | `GET /v1/models` | 模型列表 |
| **Anthropic** | `POST /v1/anthropic/messages` | Messages API |
| | `POST /v1/anthropic/messages/stream` | 流式 Messages |
| **Gemini** | `POST /v1/gemini/generateContent` | 内容生成 |
| | `POST /v1/gemini/streamGenerateContent` | 流式生成 |
| **WebSocket** | `POST /v1/ws` | 流式对话 WebSocket 代理 |

### 11.2 请求转换流程

```
Client Anthropic/Gemini format → gateway 协议检测（URL前缀）
  → 归一化为 StandardRequest（OpenAI 兼容格式）
  → 中间件链复用（认证/限流/合规/路由）
  → adapter litellm.acompletion(model="anthropic/..." 或 "gemini/...")
  → LiteLLM 自动转换为目标供应商原生格式
  → 响应时根据 protocol_origin 反向转换为客户端期望格式
```

### 11.3 阶段策略

- **MVP（当前）**：OpenAI 格式完整支持；Anthropic/Gemini 轻量映射（核心字段映射到 StandardRequest）
- **GA（P2）**：全部三种格式完整支持双向转换，`protocol` 字段注入 StandardRequest
