## 3. 关键特性实现细节补充

### 3.1 策略仿真引擎（routing-service）

当前设计仅提了"回放历史请求"，需要补充：

```
仿真引擎核心流程：
  1. 快照创建：锁定当前路由策略、端点健康状态、定价快照
  2. 请求选取：从 ClickHouse trace 表选取最近 N 条（默认 5000）实际请求
  3. 候选生成：对新策略规则生成每个请求的候选列表 + 评分
  4. 结果对比：
      - 成本变化：Σ(new_cost) - Σ(old_cost)
      - 延迟变化：P95(new_latency) - P95(old_latency)
      - 成功率变化：模拟 failover 场景
      - 供应商分布偏移
  5. 报告生成：输出对比报表 + "建议接受/拒绝" 置信评分
  6. 覆盖率要求：N ≥ 1000 且覆盖所有活跃 LogicalModel

仿真引擎状态机：
  pending → running → completed | failed → report_generated
```

### 3.2 成本异常检测（billing-service）

当前设计仅 3σ，需要补充：

```
季节性调整（STL 分解）：
  observed = trend + seasonal + residual
  异常判定：residual > 3σ_residual

异常归因分析：
  1. 维度钻取：按 project / model / vendor / hour 逐步下钻
  2. 变化点检测：PELT (Pruned Exact Linear Time) 算法
  3. 变化类型分类：
      - 水平移位（sudden level shift）
      - 趋势变化（trend change）
      - 季节性变化（seasonal change）

告警抑制：首次触发后 2h 内同维度不再重复告警
```

### 3.4 语义缓存失效策略（adapter-service）

当前仅 TTL 过期，需要增补：

```
主动失效机制：
  1. 供应商模型更新事件（Kafka maas.model.events）
     触发条件：provider_model 版本变更或定价变更
     动作：清空该 logical_model_id 的所有缓存
  2. 租户配置变更
     触发条件：zero_retention 变更、cache_enabled 变更
     动作：清空该 tenant_id 级别缓存
  3. 手动失效 API
     POST /api/v1/cache/invalidate?tenant_id=xxx&logical_model=maas:gpt-4o

过期策略：
  - 正常场景：TTL=4h（temperature=0 确定性输出）
  - 高频访问：若 1h 内命中 > 10 次，自动延长 TTL 至 24h
  - 空间淘汰：Redis maxmemory-policy=allkeys-lru
  - Qdrant：Payload 中存储 access_count + last_access_at，定时清理低价值向量
```

### 3.5 内容安全策略执行时序（compliance-service）

当前设计中各安全策略的执行顺序未明确定义：

```
完整执行流水线（gateway + compliance 协作）：

阶段 1 (Gateway 同步, P99 < 2ms):
  1. 关键词过滤器（本地词库，严重违规直接 BLOCK）
  2. 自定义 CEL 规则（轻量表达式）

阶段 2 (Gateway → Compliance 异步并行):
  3. PII 检测（Regex + NER 混合）
  4. 内容分类模型（违规类别检测）

阶段 3 (响应后 Compliance 异步):
  5. 内容分类结果写入 trace
  6. 违规事件 → Kafka maas.compliance.events

结果聚合逻辑：
  - 任一阶段返回 BLOCK → 立即拒绝（短路）
  - PII 检测 + 内容分类同时返回 WARN → 提升为 BLOCK
  - 白名单机制：tenant_id + user_id 可绕过部分 WARN 级别规则
```

## 5. 新增审核发现（2026-05-25）

> **审核人**：Claude Code（kimi-k2.6）  
> **审核范围**：gateway / routing / model-catalog / adapter / billing / llmops-trace 六个核心服务  
> **基准**：MaaS-PRD-V2.0 + 当前微服务设计文档

### 5.1 架构层面（跨服务协作）

#### 问题 1：流式响应链路存在根本性矛盾 —— SSE 穿越两层 gRPC
**等级：P0**

当前链路：gateway (HTTP) → routing (gRPC stream) → adapter (gRPC stream) → 供应商 (HTTP SSE)。SSE 响应要经历 gateway→routing→adapter 两层 gRPC stream 转换，导致：
- backpressure 失效：客户端消费慢时无法反向传播背压到 adapter；
- 客户端断开检测困难：`request.is_disconnected()` 只在 gateway HTTP 层有效，gRPC stream 无法感知原始客户端断开；
- TTFB 延迟叠加：每层协议转换增加首 Token 延迟，难以达成 "≤ 10ms" 目标。

**建议**：流式请求改为 gateway 直接 HTTP 调用 adapter-service，routing-service 只负责同步路由决策（返回 backend_id + route_explanation）。

#### 问题 2：routing-service 与 adapter-service 的缓存职责边界矛盾
**等级：P1**

routing-service 在时序图中直接查询语义缓存，但 `StandardRequest` protobuf 未暴露 temperature/stream 字段；同时 routing L5 fallback 的 "语义缓存最近命中响应" 与 adapter 的 "仅 temperature=0 可缓存" 条件冲突，导致 L5 fallback 实际上不可命中。

**建议**：明确语义缓存查缓存职责放在 routing-service（需解析 body 中的 temperature）或 adapter-service；L5 fallback 的 stale 缓存建议仅用于 embedding 等只读 API。

#### 问题 3：Key 池管理职责分裂，缺少同步机制
**等级：P2**

model-catalog-service 管理 `key_pool_ref`（配置侧），adapter-service 管理 Redis `keypool:*`（运行时侧）。缺少配置变更到运行时的同步事件。

**建议**：`maas.model.events` 中增加 `key_pool_changed` 事件，adapter-service 消费并同步 Redis。

### 5.2 adapter-service 中 LiteLLM 的使用问题

#### 问题 4：LiteLLM Router 与 routing-service 决策冲突
**等级：P0**

routing-service 已通过四维评分选定了 `backend_id`，但 adapter-service 内部的 `litellm.Router` 配置了 `routing_strategy="latency-based"`，且 `model_list` 包含多个 backend。Router 会自行在模型池内做负载均衡，可能把请求发到 routing-service 未选定的 backend，导致路由决策被覆盖。

**建议**：每次调用时临时构造只含单个目标 backend 的 `model_list`，或将 `routing_strategy` 设为 `"simple-shuffle"` 但仅放单个模型；更推荐的做法是直接调用 `litellm.acompletion()`（不经过 Router），因为 KeyPool / RetryEngine / CircuitBreaker 已自研。

#### 问题 5：`litellm.Router` 内部 API 兼容风险
**等级：P1**

设计依赖了多个未经验证的 LiteLLM 内部 API：
- `await router.update_model_list(model_list)` — 热更新方法在 1.40+ 稳定性需确认；
- `context_length_manager="ljm"` — 非 Router 标准构造参数；
- `await router.close()` — Router 是否提供 async close 需确认；
- `await router.disable_model(backend_id)` — 同样需要确认。

**建议**：在源码走读报告中明确验证上述 API；或降低对 Router 的依赖，改用 `acompletion()` 核心函数。

#### 问题 6：Prompt Cache（上下文缓存）完全缺失
**等级：P1**

Anthropic Prompt Caching、OpenAI cached_tokens、Gemini Context Caching 等平台核心降本功能在 adapter-service 中未设计。

**建议**：在 `client.py` 中增加 `cache_control` / `prompt_cache` 参数透传；billing-service 已有 `CachedTokenDiscount`，需 adapter 打通 `cached_input_tokens` 的识别与返回。

#### 问题 7：LiteLLM 异常覆盖不全
**等级：P1**

`client.py` 未捕获 `APIConnectionError`（DNS/TCP/SSL 失败）和 `InternalServerError`（供应商 5xx），这两类是生产环境最常见的错误。

**建议**：补全异常映射表。

### 5.3 实现层面代码缺陷

#### 问题 8：SSEStreamWrapper 有严重 bug
**等级：P0**

`src/litellm_integration/streaming.py` 存在三处 bug：
1. `_producer` 无 return 值，但 `event_generator` 中用 `task.result()` 判断 Exception，此分支永远不会进入；
2. `_heartbeat` 生成器每次 `__anext__()` 只取一次值后重新创建 task，导致心跳无法按 15s 周期持续发送；
3. 生产者异常时，except 分支和 finally 分支分别向队列 put Exception 和 None，队列内出现双标记，消费者未处理。

**建议**：重写 SSEStreamWrapper，简化实现，利用 FastAPI StreamingResponse 原生背压。

#### 问题 9：CircuitBreaker 实现不完整
**等级：P0**

`circuit_breaker.py` 中：
- `is_allowed()` 在 OPEN 状态下直接返回 False，但未记录 OPEN 时间戳，无法实现 `recovery_seconds` 后的 HALF_OPEN 恢复；
- 状态机只实现了 CLOSED → OPEN，缺少 HALF_OPEN → CLOSED 的完整链路；
- 滑动窗口清空后不会自动清除 OPEN 状态。

**建议**：补全状态机，记录 `opened_at` 时间戳，增加 HALF_OPEN 探测逻辑。

#### 问题 10：gRPC proto 定义有语法错误
**等级：P2**

`rpc InvalidateCache:(InvalidateRequest)` 冒号位置错误；`rpc GetKeyPoolStatus(KeyPoolStatus\Request)` 含转义符；字段对齐混乱。

**建议**：修正 proto 语法，确保可编译。

#### 问题 11：Prometheus 指标定义有语法错误
**等级：P2**

`adapter_stream_duration_seconds` 构造函数参数位置与关键字混用；`adapter_key_pool_size` 变量赋值缺失。

**建议**：修正为 `prometheus_client` 标准构造方式。

### 5.4 性能与运维风险

#### 问题 12：Embedding 计算目标 5ms 不现实
**等级：P1**

bge-m3 在 CPU 上单条推理通常 20-100ms，且 Embedding 计算是 CPU 密集型，会阻塞 asyncio 事件循环。

**建议**：目标放宽至 50ms；计算放入线程池 `loop.run_in_executor`；或独立 GPU embedding 服务。

#### 问题 13：重试 + Fallback 级联超时风险
**等级：P1**

adapter 内部重试 3 次（最大退避 10s × 3 + 供应商超时 60s），routing fallback 链最多遍历 5 层，最坏情况单请求可超过 10 分钟。routing 对 adapter 的 circuit breaker timeout 仅 30s，但 adapter 内部超时可能远超此值。

**建议**：adapter 设置总超时预算（如 15s）；routing 调用 adapter 超时小于 circuit breaker timeout；adapter 内部重试时递减剩余超时（deadline propagation）。

#### 问题 14：并发流内存余量紧张
**等级：P2**

500 并发流 × 10MB = 5GB，加上 bge-m3 缓存 1.5GB + runtime 约 1GB，8Gi limit 余量不足，易 OOMKilled。

**建议**：limit 提升至 16Gi；HPA 增加内存指标。
