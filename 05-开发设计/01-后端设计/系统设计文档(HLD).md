## 1. 系统概述

### 1.1 系统目标

MaaS（模型即服务）平台是一个企业级大模型聚合与管理平台，旨在：

- 提供**统一API网关**，兼容OpenAI协议，屏蔽多厂商模型差异
- 实现**模型全生命周期管理**：接入→精调→评估→部署→监控→下线
- 通过**智能路由引擎**在成本与性能之间动态寻优
- 提供**精细化计费与成本管控**能力
- 保障**企业级安全合规**（等保、数据不出域、RBAC审计）

### 1.2 系统边界

**系统内部（In Scope）：**
- 统一API网关与协议适配层
- 模型管理与精调工作台
- 智能路由与负载均衡引擎
- 语义缓存层
- 计量计费系统
- 监控告警中心
- 开发者控制台（Web前端）
- 多租户RBAC权限管理

**系统外部（Out of Scope）：**
- 大规模分布式模型全量训练平台
- 边缘模型管理与联邦学习
- 原生AI应用商店运营系统

## 2. 设计原则与约束

### 2.1 核心设计原则

| 原则 | 说明 |
|------|------|
| **高可用优先** | API网关SLA ≥ 99.95%，采用多活冗余架构 |
| **弹性伸缩** | 基于Kubernetes HPA/VPA，应对请求量突增 |
| **安全合规** | 零信任网络，全链路加密，操作日志不可篡改 |
| **可观测性** | Metrics / Tracing / Logging 三支柱，全栈可观测 |
| **开放兼容** | 对外API兼容OpenAI标准，对内适配器插件化 |
| **数据驱动** | 埋点先行，指标可量化，路由策略可A/B验证 |

### 2.2 技术约束

1. 必须兼容国产芯片（昇腾 NPU、海光 DCU）与操作系统环境
2. 私有化部署时，模型数据与业务数据不出本地数据中心
3. 核心推理延迟 P95 ≤ 800ms（不含网络传输）
4. 必须支持 GPU 异构调度（NVIDIA、昇腾混合集群）

```

         ↓（异步）
         日志采集 → Kafka → 计量计费 + 监控指标
```

## 4. 核心子系统设计

### 4.1 统一API网关

**职责：** 作为平台的统一流量入口，负责认证、限流、协议转换、日志埋点。

**核心模块：**

| 模块 | 功能 | 技术实现 |
|------|------|---------|
| 认证模块 | API Key验证、JWT校验、IP白名单 | 自研 + Redis缓存 |
| 限流模块 | RPM/TPM/并发数多维限流 | Nginx + Lua / 令牌桶算法 |
| 协议转换 | OpenAI格式 → 各厂商原生格式 | 插件化Adapter Pattern |
| 日志埋点 | 请求/响应全量记录，异步写入 | Filebeat → Kafka |
| 熔断模块 | 下游服务异常时快速失败 | Hystrix / Sentinel |

**关键接口：**
- `POST /v1/chat/completions` - 对话补全（OpenAI兼容）
- `POST /v1/embeddings` - 文本嵌入
- `POST /v1/completions` - 文本补全（Legacy）
- `GET /v1/models` - 可用模型列表

**非功能指标：**
- QPS处理能力：≥ 10,000 req/s（集群级别）
- 网关自身额外延迟：< 10ms（P99）
- 高可用：双活部署，单节点故障30秒内切换

### 4.2 智能路由引擎

**职责：** 根据路由策略、实时指标和成本模型，将请求分发至最优模型实例。

**路由决策流程：**

```
请求到达 → 提取路由Key（api_key_id + model_hint）
         → 加载路由策略（优先级排序）
         → 过滤候选模型（健康状态、能力匹配）
         → 多维评分计算：
              score_i = w1*(1/cost_i) + w2*(1/latency_i) + w3*match_i + w4*stability_i
         → 选择最高得分实例
         → 执行调用，监控结果
         → 更新实例统计（EWMA）
```

**路由策略类型：**

| 策略类型 | 适用场景 | 说明 |
|---------|---------|------|
| 成本优先 | 批量处理、非实时任务 | 选择单位Token成本最低模型 |
| 性能优先 | 实时交互、低延迟要求 | 选择P95延迟最低实例 |
| 指定模型 | 特定场景定制化调用 | 直接路由至指定模型标识 |
| 轮询均衡 | 同质实例间均衡 | 加权轮询算法 |
| A/B测试 | 策略验证阶段 | 按流量比例分割 |

### 4.3 语义缓存层

**两级缓存架构：**

```
L1（精确匹配）: MD5(normalize(prompt)) → Redis String, TTL=24h
L2（语义匹配）: Embedding向量 → Milvus 向量检索, Top-K余弦相似度 > 0.92 命中
```

**嵌入模型：** 使用内置轻量嵌入模型（如 bge-m3），避免缓存查询本身产生大量Token消耗。

**缓存更新策略：**
- 写后立即建立L1缓存及异步建立L2索引
- L1 LRU驱逐，最大容量可配置（默认4GB）
- L2定期清理低热度向量（访问频次 < 阈值）

### 4.4 模型生命周期管理

**状态机：**

```
[已接入] → [体验中] → 用户测试完毕
         → [已精调] → [评估中] → [已就绪] → [部署中] → [运行中] → [监控中]
                               → [精调失败]                    → [异常] → [已下线]
[已下线] → 归档
```

**关键服务：**

| 服务 | 职责 |
|------|------|
| 模型注册服务 | 接入供应商模型、录入元数据（名称、能力标签、计费单价） |
| 精调任务调度器 | 管理LoRA/QLoRA微调任务队列，对接GPU资源池 |
| 模型评估服务 | 基于标准评测集（MMLU、HumanEval等）自动评估 |
| 模型部署管理器 | 对接K8s，将模型容器化部署为推理服务 |
| 健康检查服务 | 定时探活，维护实例健康状态 |

### 4.5 计量计费引擎

**计费流程：**

```
API调用完成 → 异步发送计费事件至Kafka
           → 计费消费者读取事件
           → 查询模型实时单价（输入Token价 / 输出Token价）
           → 计算本次调用费用
           → 写入账单明细表（PostgreSQL）
           → 按api_key/project/tenant 实时累计使用量
           → 检查预算预警阈值 → 触发告警
```

**计费精度要求：** 误差率 < 0.01%（计费关键路径，采用幂等写入保障一致性）

## 5. 数据架构设计

### 5.1 数据存储选型

| 数据类型 | 存储方案 | 理由 |
|---------|---------|------|
| 业务主数据（用户、项目、模型元数据） | PostgreSQL | ACID事务，关系完整性 |
| 请求日志明细 | Elasticsearch | 全文检索，大规模日志查询 |
| 实时监控指标 | InfluxDB / Victoria Metrics | 时序数据，高写入吞吐 |
| 语义向量缓存 | Milvus | ANN向量检索 |
| 缓存与会话 | Redis Cluster | 高性能键值，支持TTL |
| 消息队列 | Apache Kafka | 日志/事件解耦，高吞吐 |
| 对象存储（数据集/模型文件） | MinIO / OSS | 大文件存储 |

### 5.2 核心数据实体关系（概念层）

```
Tenant（租户）
  ├── User（用户）
  │    └── ApiKey（密钥）
  ├── Project（项目）
  │    ├── ApiKey（密钥）
  │    ├── RoutingPolicy（路由策略）
  │    └── BudgetConfig（预算配置）
  ├── ModelInstance（模型实例）
  │    ├── ModelVersion（模型版本）
  │    └── FineTuneJob（精调任务）
  └── BillingStatement（账单）
       └── UsageRecord（用量明细）
```

## 6. 部署架构设计

### 6.1 生产环境部署拓扑

```
公网流量 → CDN/WAF → 负载均衡器 (LB)
                         │
         ┌───────────────┼───────────────┐
         ▼               ▼               ▼
    API网关Pod1      API网关Pod2      API网关PodN
         └───────────────┼───────────────┘
                         │（K8s Service）
         ┌───────────────┼───────────────┐
         ▼               ▼               ▼
    路由引擎Pod1     路由引擎Pod2    路由引擎PodN
                         │
         ┌───────────────┼───────────────┐
         ▼               ▼               ▼
    模型适配器集群    缓存服务集群    计费服务集群
                         │
    ┌────────────────────┴────────────────────┐
    ▼                                         ▼
GPU推理集群（自托管模型）           外部模型API（云厂商）
```

### 6.2 高可用设计

| 组件 | 冗余方案 | RTO | RPO |
|------|---------|-----|-----|
| API网关 | 多Zone双活，≥3副本 | < 30s | 0 |
| 路由引擎 | 无状态，多副本 | < 30s | 0 |
| PostgreSQL | 主从复制 + 自动Failover | < 60s | < 5s |
| Redis | Cluster模式，3主3从 | < 30s | 0（内存） |
| Kafka | 多副本分区，RF=3 | < 60s | 接近0 |

### 6.3 弹性伸缩策略

- **API网关**：HPA基于CPU使用率（> 70%扩容），最小3副本，最大20副本
- **路由引擎**：HPA基于请求队列长度，响应时间P95超阈值触发扩容
- **GPU推理节点**：基于GPU使用率 + 请求队列，异构节点池调度（NVIDIA优先，昇腾备用）

## 7. 安全架构设计

### 7.1 安全分层防护

```
外部威胁防护：WAF（SQL注入/XSS）、DDoS防护、速率限制
认证与授权：API Key认证、RBAC权限控制、最小权限原则
数据安全：传输TLS 1.3加密、静态数据AES-256加密、PII自动脱敏
网络隔离：K8s NetworkPolicy隔离、服务间mTLS通信
审计追踪：操作日志不可篡改（Append-Only）、全链路请求追踪
```

### 7.2 内容安全机制

1. **请求过滤层**：在网关层嵌入内容安全检测，拦截违规Prompt
2. **响应过滤层**：对模型输出进行二次安全扫描（可配置敏感词库）
3. **分级策略**：可按Tenant/Project配置不同严格程度的安全策略

### 7.3 数据隐私合规

- 请求日志中的用户Prompt默认自动脱敏（姓名、手机号、身份证等PII）
- 支持私有化部署模式，数据完全不出客户数据中心
- 用户数据保留策略可配置，满足GDPR和数据安全法要求

## 8. 非功能性需求设计

### 8.1 性能目标

| 指标 | 目标值 | 测量方式 |
|------|--------|---------|
| API网关可用性 | ≥ 99.95% | Uptime监控 |
| 请求P95延迟（不含模型推理） | ≤ 50ms | APM追踪 |
| 请求P95延迟（含模型推理） | ≤ 800ms | 端到端监控 |
| 峰值QPS（集群） | ≥ 10,000 | 压力测试 |
| 语义缓存命中率 | ≥ 20% | 埋点统计 |
| GPU资源利用率 | ≥ 70% | 基础设施监控 |

### 8.2 可观测性设计

**三支柱体系：**

| 支柱 | 工具 | 用途 |
|------|------|------|
| **Metrics** | Prometheus + Grafana | 系统与业务指标实时监控 |
| **Tracing** | Jaeger / OpenTelemetry | 分布式链路追踪，定位跨服务延迟 |
| **Logging** | ELK Stack（ES+Logstash+Kibana） | 日志聚合、查询、告警 |

**告警层级：**

| 等级 | 触发条件示例 | 通知方式 | 响应时间 |
|------|-------------|---------|---------|
| P0 - 严重 | 网关不可用 / 数据泄露 | 电话 + 短信 + 钉钉 | 15分钟内 |
| P1 - 高 | 错误率 > 1% / 延迟P99 > 3s | 短信 + 钉钉 | 30分钟内 |
| P2 - 中 | 缓存命中率骤降 / 预算告警 | 钉钉 + 邮件 | 2小时内 |
| P3 - 低 | 日报异常 / 容量预测告警 | 邮件 | 下个工作日 |

## 9. 关键技术选型

| 层次 | 组件 | 选型 | 选型理由 |
|------|------|------|---------|
| API网关 | 流量接入 | Nginx + 自研网关服务 | 高性能，可扩展，支持插件 |
| 路由引擎 | 核心服务 | Go（高并发，低延迟） | GC开销低，并发模型优异 |
| 模型管理 | 推理框架 | vLLM / TGI | 高吞吐量推理，PagedAttention优化 |
| 精调框架 | 微调工具 | LLaMA-Factory / PEFT | 支持LoRA/QLoRA，多模型适配 |
| 容器编排 | 调度平台 | Kubernetes + GPU Operator | 行业标准，支持GPU异构调度 |
| 消息队列 | 异步解耦 | Apache Kafka | 高吞吐，持久化，多消费者组 |
| 向量数据库 | 语义缓存 | Milvus | 开源，高性能ANN检索 |
| 前端框架 | 控制台 | React + TypeScript | 生态成熟，组件库丰富 |
| CI/CD | 流水线 | GitLab CI + ArgoCD | GitOps，K8s原生部署 |

## 10. 系统接口设计

### 10.1 外部接口（对开发者）

遵循 OpenAI API 标准，主要端点：

| 接口 | 方法 | 说明 |
|------|------|------|
| `/v1/chat/completions` | POST | 对话补全，支持Streaming |
| `/v1/completions` | POST | 文本补全（Legacy） |
| `/v1/embeddings` | POST | 文本嵌入向量 |
| `/v1/models` | GET | 列出可用模型 |
| `/v1/models/{model_id}` | GET | 查询模型详情 |

### 10.2 内部服务间接口

- **网关 → 路由引擎**：gRPC（`RoutingService.Route`），传递请求上下文
- **路由引擎 → 适配器**：HTTP/2，传递标准化请求体
- **适配器 → 后端模型**：各厂商原生REST API（通过Adapter抽象）
- **各服务 → Kafka**：异步事件发布（日志、计费事件、状态变更）
- **各服务 → Prometheus**：Pull模式暴露 `/metrics` 端点

### 10.3 外部系统集成接口

| 外部系统 | 集成方式 | 数据内容 |
|---------|---------|---------|
| 公司SSO系统 | OAuth2 / SAML | 用户认证与身份同步 |
| 财务ERP | REST API + 定时同步 | 账单数据、费用归因 |
| 基础设施算力平台 | K8s API + 自定义Operator | GPU资源调度与监控 |
| 钉钉/企业微信 | Webhook | 告警通知推送 |
