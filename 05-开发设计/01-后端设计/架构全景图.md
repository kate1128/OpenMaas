## 1. 宏观分层架构

```
┌─────────────────────────────────────────────────────────────────────┐
│  L0 接入层    │ 开发者 SDK │ CLI 工具 │ Web Console │ 外部 SSO/SIEM │
│              │ 协议: OpenAI / Anthropic / Gemini REST + WebSocket    │
├─────────────────────────────────────────────────────────────────────┤
│  L1 网关层    │ gateway-service (Go)                                 │
│              │ 11步中间件链: 协议检测→认证→限流→预算→合规→安全→标准化  │
│              │ →路由分发→供应商调用→响应标准化→事件埋点               │
├─────────────────────────────────────────────────────────────────────┤
│  L2 路由层    │ routing-service (Go)                                 │
│              │ 四维评分 │ 5策略类型 │ L1-L5 Fallback │ A/B联动       │
├──────────────┬──────────────────────────────────────────────────────┤
│  L3 执行层   │ adapter-service (Python + LiteLLM)                    │
│              │ 100+供应商协议翻译 │ Key池+熔断 │ 语义缓存 │ 流式代理  │
├──────────────┴──────────────────────────────────────────────────────┤
│  L4 业务层   │ model-catalog-service │ billing-service              │
│              │ llmops-trace-service  │ prompt-eval-service           │
│              │ auth-service          │ compliance-service             │
│              │ notification-service                                   │
├─────────────────────────────────────────────────────────────────────┤
│  L5 消息层   │ Apache Kafka (7个核心 Topic)                          │
├─────────────────────────────────────────────────────────────────────┤
│  L6 存储层   │ PostgreSQL │ ClickHouse │ Redis Cluster │ Qdrant     │
│              │ OSS/MinIO │ Elasticsearch                             │
└─────────────────────────────────────────────────────────────────────┘
```

## 3. 请求处理全链路时序图

### 3.1 同步请求（非流式）

```mermaid
sequenceDiagram
    autonumber
    participant Client as 开发者
    participant GW as gateway-service
    participant Auth as auth-service
    participant Billing as billing-service
    participant Compliance as compliance-service
    participant Routing as routing-service
    participant Adapter as adapter-service
    participant Vendor as 外部供应商API
    participant Kafka as Kafka

    Client->>GW: POST /v1/chat/completions<br/>Authorization: Bearer mk-prod-xxx

    %% 中间件链
    GW->>GW: 0.生成 trace_id / request_id
    GW->>GW: 1.协议检测(OpenAI格式)

    GW->>GW: 2.认证: HMAC-SHA256(key, secret)
    GW->>Auth: gRPC ValidateApiKey(缓存未命中时)
    Auth-->>GW: ApiKeyContext{tenant, project, scope}

    GW->>GW: 3.RPM/TPM限流(Redis令牌桶)
    GW->>Billing: 4.gRPC PreCheckQuota(tenant_id, est_tokens)
    Billing-->>GW: ALLOW
    GW->>Compliance: 5.gRPC CheckDataResidency(tenant_id, model_id)
    Compliance-->>GW: ALLOW

    GW->>GW: 6.本地内容安全预检(P99<2ms)
    GW->>GW: 7.请求标准化(Schema校验+字段补全)

    GW->>Routing: 8.gRPC RouteRequest(StandardRequest)
    Routing->>Routing: 四维评分+策略匹配
    Routing-->>GW: RouteDecision{backend_id, fallback_chain}

    GW->>Adapter: 9.gRPC CallVendorBackend
    Adapter->>Adapter: KeyPool选择可用Key
    Adapter->>Adapter: CircuitBreaker检查
    Adapter->>Vendor: LiteLLM HTTP调用<br/>(协议自动转换)
    Vendor-->>Adapter: 模型响应 + usage字段
    Adapter-->>GW: VendorCallResponse

    GW->>GW: 10.响应标准化+注入x_maas扩展头
    GW-->>Client: HTTP 200 + 响应体

    GW->>Kafka: 11.异步: api_request_completed事件
```

### 3.2 流式请求（SSE）

```mermaid
sequenceDiagram
    autonumber
    participant Client as 开发者
    participant GW as gateway-service
    participant Routing as routing-service
    participant Adapter as adapter-service
    participant Vendor as 外部供应商API
    participant Kafka as Kafka

    Client->>GW: POST /v1/chat/completions<br/>(stream=true)

    GW->>GW: 认证/限流/预算/合规/安全/标准化(同同同步)
    GW->>Routing: gRPC RouteRequest(stream=true)
    Routing-->>GW: RouteDecision{backend_id}

    Note over GW,Adapter: ★关键架构决策★<br/>流式不走gRPC代理<br/>gateway直接HTTP调用adapter

    GW->>Adapter: HTTP POST /v1/chat/completions/stream
    Adapter->>Adapter: KeyPool + CircuitBreaker
    Adapter->>Vendor: LiteLLM acompletion(stream=True)

    loop SSE流式传输
        Vendor-->>Adapter: SSE chunk
        Adapter->>Adapter: SSEStreamWrapper(心跳+背压)
        Adapter-->>GW: SSE chunk (透传)
        GW-->>Client: SSE chunk (透传)
    end

    Adapter-->>GW: 流结束: [DONE] + usage
    GW-->>Client: [DONE]
    GW->>Kafka: api_request_completed事件
```

## 5. 路由决策四维评分

```mermaid
flowchart TD
    REQ["StandardRequest<br/>到达routing-service"] --> SCOPE

    subgraph SCOPE["1. 四级作用域解析"]
        direction LR
        PLAT["平台级策略<br/>(全局默认)"] --> TEN["租户级策略<br/>(override)"] --> PROJ["项目级策略<br/>(override)"] --> KEY["Key级策略<br/>(override)"]
    end

    SCOPE --> MATCH

    subgraph MATCH["2. 策略匹配"]
        direction TB
        M1["COST_OPTIMAL<br/>最低成本优先"] 
        M2["PERFORMANCE<br/>最低延迟优先"]
        M3["FIXED_MODEL<br/>指定backend_id"]
        M4["WEIGHTED_RR<br/>加权轮询"]
        M5["CANARY<br/>A/B实验分流"]
    end

    MATCH --> FILTER

    subgraph FILTER["3. 候选过滤"]
        direction LR
        F1["健康过滤<br/>health_score>0"] --> F2["能力过滤<br/>capability_match>0"] --> F3["区域过滤<br/>data_residency合规"] --> F4["熔断过滤<br/>CircuitBreaker=CLOSED"]
    end

    FILTER --> SCORE

    subgraph SCORE["4. 四维评分"]
        direction TB
        S1["W₁×cost_score<br/>成本越低分越高<br/>cost_score=1/(price×est_tokens+ε)"]
        S2["W₂×latency_score<br/>延迟越低分越高<br/>latency_score=1/(P95_latency+1)"]
        S3["W₃×capability_match<br/>能力标签匹配率<br/>∩(required, provided)/required"]
        S4["W₄×health_score<br/>五维健康评分<br/>0.3×成功率+0.2×延迟+0.2×限流+0.2×配额+0.1×事故"]
    end

    SCORE --> DECISION

    DECISION["5. RouteDecision<br/>{backend_id, score,<br/>fallback_chain[L2..L5],<br/>route_explanation}"] --> RESP["返回gateway"]

    FILTER -->|"无可用候选"| FALLBACK

    subgraph FALLBACK["6. Fallback链"]
        direction LR
        FB["L1: 主后端<br/>(最高分)"] --> FB2["L2: 同vendor其他后端"]
        FB2 --> FB3["L3: 跨vendor同档模型<br/>(cap≥0.8)"]
        FB3 --> FB4["L4: 跨vendor降级模型<br/>(cap≥0.6)"]
        FB4 --> FB5["L5: 缓存兜底<br/>(静态fallback)"]
    end

    style SCORE fill:#e0f0ff
    style DECISION fill:#e0ffe0
    style FALLBACK fill:#ffe0e0
```

**四维权重默认值：** `W₁=0.30` `W₂=0.35` `W₃=0.20` `W₄=0.15`（可通过策略配置覆盖）

**PII 检测两阶段混合策略：**

## 10. 预扣-核销计费流

```mermaid
sequenceDiagram
    autonumber
    participant Client as 开发者
    participant GW as gateway
    participant BS as billing-service
    participant Redis as Redis
    participant Adapter as adapter
    participant Vendor as 供应商
    participant Kafka as Kafka

    Note over Client,Vendor: ═══ 第一阶段：预扣（请求前，P99<5ms） ═══
    Client->>GW: POST /v1/chat/completions
    GW->>BS: gRPC PreCheckQuota(tenant_id, est_tokens)
    Note over BS: est_tokens = prompt(已知) + completion_upper_bound
    Note over BS: completion_upper_bound = max_tokens(优先)<br/>或 context_window×0.5(兜底)
    BS->>BS: 预估费 = est_tokens × max(输入单价, 输出单价)
    BS->>Redis: GET billing:balance:{tenant_id}
    BS->>BS: 检查四层预算(平台/租户/项目/Key)
    BS-->>GW: ALLOW (仅放行，不扣费)

    Note over Client,Vendor: ═══ 第二阶段：请求执行 ═══
    GW->>Adapter: 转发请求
    Adapter->>Vendor: LiteLLM调用

    Note over Client,Vendor: ═══ 第三阶段：核销（响应后） ═══
    Vendor-->>Adapter: 响应 + usage字段
    Adapter-->>GW: VendorCallResponse
    GW-->>Client: 返回结果
    GW->>Kafka: api_request_completed事件

    Kafka->>BS: 消费
    BS->>BS: 幂等检查(request_id去重)
    BS->>BS: 精确计算: prompt_tokens×输入价 + completion_tokens×输出价
    BS->>BS: + cached_input_tokens×0.1×输入价
    BS->>BS: - discount_amount
    BS->>BS: = net_amount (实际扣费)
    BS->>Redis: DECR billing:balance:{tenant_id}
    BS->>BS: 检查预算阈值(50%/80%/100%)
    opt 超阈值
        BS->>Kafka: budget_threshold_reached → notification
    end
```

**三优先级计量：**
```
1️⃣ 供应商 usage 字段（最准，优先使用）
2️⃣ 网关侧 Tokenizer 计算（供应商未返回时）
3️⃣ 字符估算 = 字符数/4（兜底，is_estimated=true）
```
