# 灾难恢复演练手册（DR Drill）

**文档版本：** V1.0  
**编写日期：** 2026年05月14日  
**演练频率：** 每季度一次全量演练，每月一次专项演练  
**负责人：** SRE 团队负责人  
**参与人员：** SRE 全员 + 各服务 Owner

> **注意：** 本手册为主动演练脚本（模拟故障注入）。故障响应请参见《运维手册(Runbook).md》。

---

## 1. 演练目标与原则

### 1.1 目标

- 验证 RTO/RPO 目标可达性
- 发现 Runbook 中的流程缺陷
- 提高团队在真实故障下的响应熟练度
- 测试监控告警的灵敏性与准确性

### 1.2 RTO/RPO 目标

| 组件 | RTO（恢复时间目标） | RPO（恢复点目标） | 演练验证方式 |
|------|-----------------|----------------|------------|
| API 网关 | < 5 分钟 | N/A（无状态） | 模拟 Pod 全量崩溃 |
| PostgreSQL 主库 | < 15 分钟 | < 1 分钟（WAL） | 主库强制 Failover |
| Redis Cluster | < 10 分钟 | < 30 秒 | 主节点故障注入 |
| Kafka Cluster | < 10 分钟 | < 30 秒（副本同步） | Leader 重选举 |
| 整个可用区（AZ） | < 30 分钟 | < 5 分钟 | 流量切换演练 |
| 整个 Region | < 4 小时 | < 15 分钟 | 异地 DR 演练（每半年） |

### 1.3 演练原则

```
1. 生产环境优先在低峰期（周日 02:00-06:00）执行
2. 演练前必须通知所有值班人员
3. 每次演练设置明确的"终止条件"——超出预期影响立即停止
4. 演练结束后 24 小时内输出演练报告
5. 所有演练操作必须记录到演练日志
```

### DR-01：单个微服务全量崩溃

**目标：** 验证 K8s 自愈和 HPA 的恢复能力  
**演练时长：** 30 分钟  
**告警预期：** 服务不可用告警（P1）在 60 秒内触发

```bash
# ===== 演练准备 =====
# 1. 记录当前状态
kubectl get pods -n maas-prod -l app=gateway-service
kubectl top pods -n maas-prod -l app=gateway-service

# 2. 确认监控大盘已就绪（Grafana 打开）

# ===== 故障注入 =====
# 3. 强制删除 gateway-service 所有 Pod（模拟进程崩溃）
kubectl delete pods -n maas-prod -l app=gateway-service --grace-period=0 --force

# 记录时间戳 T0

# ===== 观测指标 =====
# 监控以下内容：
# - 告警触发时间（预期 < 60s）
# - K8s 重新调度新 Pod 时间（预期 < 120s）
# - API 错误率恢复到 < 0.1%（预期 < 5 分钟）

# 观测命令
watch -n 2 kubectl get pods -n maas-prod -l app=gateway-service

# ===== 验证恢复 =====
# 4. 确认 Pod 恢复 Running 状态
kubectl get pods -n maas-prod -l app=gateway-service

# 5. 验证 API 可用性
curl -s https://api.maas-platform.com/health | jq .

# 6. 检查是否有请求丢失（查看 Kafka 消费 lag）
kafka-consumer-groups.sh --bootstrap-server kafka:9092 \
  --describe --group maas-billing-consumer
```

### DR-02：PostgreSQL 主库故障转移

**目标：** 验证 Patroni HA 自动 Failover 和 RTO < 15 分钟  
**演练时长：** 60 分钟  
**风险级别：** 高（涉及数据层，需主管审批）

```bash
# ===== 演练准备 =====
# 1. 确认 Patroni 集群状态健康
kubectl exec -n maas-data patroni-0 -- patronictl -c /etc/patroni/config.yml list

# 期望输出：
# + Cluster: maas-postgres +----+-----------+
# | Member     | Host       | Role   | State   |
# +------------+------------+--------+---------+
# | postgres-0 | 10.0.x.x   | Leader | running |
# | postgres-1 | 10.0.x.x   | Replica| running |
# | postgres-2 | 10.0.x.x   | Replica| running |

# 2. 记录当前主库和 LSN
kubectl exec -n maas-data patroni-0 -- psql -U postgres -c "SELECT pg_current_wal_lsn();"

# ===== 故障注入 =====
# 3. 模拟主库节点失联（暂停容器，不杀死进程）
kubectl exec -n maas-data patroni-0 -- kill -STOP 1  # 发送 SIGSTOP

# 记录时间戳 T0

# ===== 观测 Failover 过程 =====
# 4. 观察 Patroni 领导选举
watch -n 2 'kubectl exec -n maas-data patroni-1 -- \
  patronictl -c /etc/patroni/config.yml list'

# 5. 观察服务端数据库连接（连接池 PgBouncer 切换）
watch -n 2 'kubectl exec -n maas-data pgbouncer-0 -- \
  psql -p 6432 pgbouncer -c "SHOW POOLS;"'

# ===== 恢复主库 =====
# 6. 恢复原主库（变为 Replica）
kubectl exec -n maas-data patroni-0 -- kill -CONT 1  # 发送 SIGCONT

# 7. 验证原主库以 Replica 身份重新加入
kubectl exec -n maas-data patroni-1 -- \
  patronictl -c /etc/patroni/config.yml list

# ===== 数据一致性验证 =====
# 8. 验证 Failover 期间无数据丢失
kubectl exec -n maas-data patroni-1 -- psql -U postgres -c "
  SELECT count(*) FROM billing.usage_records
  WHERE created_at > NOW() - INTERVAL '10 minutes';
"
```

| 应用层感知恢复 | < 5min | ___min |
| 数据丢失记录数 | 0 | ___ |

### DR-05：单 AZ 网络中断

**目标：** 验证多 AZ 部署下，单 AZ 中断时业务自动切换  
**演练时长：** 90 分钟  
**风险级别：** 高（影响 1/3 生产容量）

```bash
# ===== 演练准备 =====
# 1. 确认三个 AZ 的 Pod 分布
kubectl get pods -n maas-prod -o wide | grep -E "node-az[1-3]"

# 2. 确认 Istio 流量分配（各 AZ 各 1/3）
kubectl get virtualservice -n maas-prod gateway-vs -o yaml

# ===== 故障注入：模拟 AZ-3 网络中断 =====
# 3. 在 AZ-3 节点上添加网络策略，切断其与其他 AZ 的通信
# （通过 K8s NetworkPolicy 或 iptables 模拟）
kubectl apply -f - <<EOF
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: dr-drill-az3-isolation
  namespace: maas-prod
spec:
  podSelector:
    matchLabels:
      topology.kubernetes.io/zone: az-3
  policyTypes:
    - Ingress
    - Egress
  ingress: []   # 拒绝所有入站
  egress: []    # 拒绝所有出站
EOF

# 记录时间戳 T0

# ===== 观测流量重新分配 =====
# 4. 观察 Istio 自动摘除不健康的 AZ-3 端点
watch -n 5 'kubectl exec -n istio-system \
  $(kubectl get pods -n istio-system -l app=pilot -o jsonpath="{.items[0].metadata.name}") \
  -- pilot-discovery request GET /debug/endpointz | jq . | grep az-3'

# 5. 观察 API 错误率（预期：短暂升高后恢复到 < 0.1%）
# 在 Grafana: maas-platform-overview 大盘

# ===== 恢复 =====
# 6. 删除网络隔离策略，恢复 AZ-3
kubectl delete networkpolicy dr-drill-az3-isolation -n maas-prod

# 7. 验证 AZ-3 Pod 重新接入流量
watch -n 2 kubectl get pods -n maas-prod -o wide
```

### DR-06：厂商 API 全量不可用

**目标：** 验证 Failover 到备用厂商的自动路由机制  
**演练时长：** 30 分钟

```bash
# ===== 通过路由配置模拟厂商不可用 =====
# 1. 将 OpenAI 所有端点标记为 unhealthy
curl -X PATCH https://admin-api.maas-platform.com/admin/v1/vendors/openai/endpoints \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"status": "maintenance", "reason": "DR drill"}'

# 2. 观察 routing-service 的 Failover 决策
kubectl logs -n maas-prod -l app=routing-service -f | grep "failover"

# 3. 发起测试请求，验证自动切换到其他厂商
for i in $(seq 1 10); do
  curl -s -X POST https://api.maas-platform.com/v1/chat/completions \
    -H "Authorization: Bearer $TEST_API_KEY" \
    -d '{"model":"gpt-4o","messages":[{"role":"user","content":"hi"}]}' \
    | jq '{model: .model, vendor: .usage.vendor}'
done

# 预期：所有请求自动 Failover 到 claude-3-5-sonnet 或 qwen-turbo

# 4. 恢复 OpenAI 端点
curl -X PATCH https://admin-api.maas-platform.com/admin/v1/vendors/openai/endpoints \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -d '{"status": "active"}'
```

### DR-08：数据库数据误删恢复

**目标：** 验证 PITR（Point-in-Time Recovery）能力  
**演练时长：** 60 分钟  
**注意：** 在独立的 DR 测试数据库上演练，禁止在生产 DB 上操作

```bash
# ===== 环境：DR 测试数据库（从生产备份恢复的副本）=====

# 1. 记录当前测试 DB 的数据量
psql -h dr-test-pg -U postgres -c "SELECT count(*) FROM billing.usage_records;"

# 2. 模拟误操作（删除今日账单记录）
TEST_TIME=$(date -u +"%Y-%m-%dT%H:%M:%SZ")
psql -h dr-test-pg -U postgres -c "
  DELETE FROM billing.usage_records
  WHERE DATE(created_at) = CURRENT_DATE;
"

# 3. 确认数据已删除
psql -h dr-test-pg -U postgres -c "
  SELECT count(*) FROM billing.usage_records
  WHERE DATE(created_at) = CURRENT_DATE;
"
# 期望：0

# 4. 使用 PITR 恢复到误删时间点之前
# （使用 Barman 或 pgBackRest）
barman recover \
  --target-time "$TEST_TIME" \
  --target-action promote \
  maas-postgres latest \
  /var/lib/postgresql/recovery

# 5. 验证数据已恢复
psql -h dr-test-pg-recovered -U postgres -c "
  SELECT count(*) FROM billing.usage_records
  WHERE DATE(created_at) = CURRENT_DATE;
"
# 期望：恢复到误删前的数量
```
