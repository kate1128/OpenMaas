## 1. Istio 选型

### 1.1 为什么选择 Istio

MaaS 架构在 V2.0 技术架构文档中将 Istio 标记为"可选，生产环境推荐"。本文档正式建议**生产环境强制启用 Istio**，原因如下：

| 需求 | 无 Mesh 时的方案 | Istio 方案 | Istio 优势 |
|------|-----------------|-----------|-----------|
| 服务间 mTLS | 应用代码实现或 VPN | 自动 Sidecar 注入 + mTLS | 零代码改动 |
| 灰度/金丝雀 | 手动修正 Deployment 比例 | VirtualService + DestinationRule | 按 Header/权重精细控制 |
| 熔断 | 应用代码实现（gobreaker） | Envoy 连接池熔断 | 无侵入，统一配置 |
| 重试 | 应用代码实现 | Envoy 自动重试 | 统一策略，减少重复代码 |
| 链路追踪 | OTel SDK 手动埋点 | Envoy 自动生成 Span | 覆盖所有 Sidecar 间流量 |
| 流量镜像 | 应用代码实现 | VirtualService mirror | 零代码 A/B 测试 |
| 多集群通信 | K8s Service 跨集群 | Istio Multi-Cluster | 统一服务发现 |

### 1.2 Istio 版本要求

| 组件 | 版本 | 说明 |
|------|------|------|
| Istio | 1.20+ | 支持 Gateway API |
| Kubernetes | 1.28+ | 与 MaaS K8s 版本一致 |
| Envoy | Istio 内置 | 1.28+ |

## 2. Sidecar 注入策略

### 2.1 注入范围

| 策略 | 范围 | 理由 |
|------|------|------|
| 全量注入 | 所有 Go 微服务 | 服务间通信全面 Mesh 化 |
| 排除注入 | `adapter-service`（Python） | Python 调试工具链与 Envoy 兼容性问题，且 adapter 主要对外 HTTP 调用供应商，服务间调用量低 |
| 可选注入 | `prompt-eval-service` | 评测执行器可能启动短生命周期 Job，Sidecar 生命周期管理复杂 |

### 2.2 Namespace 标签

```yaml
# 启用自动注入的 namespace
apiVersion: v1
kind: Namespace
metadata:
  name: maas
  labels:
    istio-injection: enabled
```

```yaml
# 单个 Pod 排除注入（adapter-service）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: adapter-service
spec:
  template:
    metadata:
      annotations:
        sidecar.istio.io/inject: "false"
```

### 2.3 Sidecar 资源分配

```yaml
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    metadata:
      annotations:
        sidecar.istio.io/proxyCPU: "100m"
        sidecar.istio.io/proxyCPULimit: "500m"
        sidecar.istio.io/proxyMemory: "128Mi"
        sidecar.istio.io/proxyMemoryLimit: "256Mi"
```

## 3. mTLS 配置

### 3.1 全网格 mTLS

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: maas
spec:
  mtls:
    mode: STRICT  # 严格要求 mTLS
```

### 3.2 选择性豁免

对于 adapter-service 的出站请求（调用外部供应商 API），不需要 mTLS：

```yaml
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: adapter-service-outbound
  namespace: maas
spec:
  selector:
    matchLabels:
      app: adapter-service
  portLevelMtls:
    8000:  # adapter-service 的 HTTP 端口
      mode: DISABLE
```

### 3.3 证书轮换

Istio 自动管理 mTLS 证书轮换（默认 24h），无需手动干预。

## 4. 流量管理

### 4.1 服务间超时

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: routing-service
spec:
  hosts:
    - routing-service
  http:
    - timeout: 5s  # 全局超时
      route:
        - destination:
            host: routing-service
```

### 4.2 重试策略

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: adapter-service
spec:
  hosts:
    - adapter-service
  http:
    - retries:
        attempts: 3
        perTryTimeout: 2s
        retryOn: connect-failure,refused-stream,unavailable,503
      route:
        - destination:
            host: adapter-service
            port:
              number: 9002  # gRPC 端口
```

### 4.3 熔断

```yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: adapter-service-circuit-breaker
spec:
  host: adapter-service
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100        # 最大连接数
      http:
        http1MaxPendingRequests: 10
        http2MaxRequests: 100
        maxRequestsPerConnection: 10
    outlierDetection:
      consecutive5xxErrors: 5      # 连续 5 次 5xx 则弹出
      interval: 30s
      baseEjectionTime: 60s
      maxEjectionPercent: 50       # 最多弹出 50% 的 Pod
```

### 4.4 灰度路由

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: routing-service-canary
spec:
  hosts:
    - routing-service
  http:
    - match:
        - headers:
            x-maas-version:
              exact: v2.1
      route:
        - destination:
            host: routing-service
            subset: v2.1
    - route:
        - destination:
            host: routing-service
            subset: v2.0
          weight: 90
        - destination:
            host: routing-service
            subset: v2.1
          weight: 10
```

## 5. 可观测性集成

### 5.1 自动 Metrics

Istio 自动采集 Envoy 指标，暴露在 Pod 的 `:15090` 端口：

```
istio_requests_total
istio_request_duration_milliseconds
istio_request_bytes
istio_response_bytes
istio_tcp_sent_bytes_total
```

Prometheus 配置：

```yaml
# Prometheus 自动发现 Istio 指标
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: istio-mesh
spec:
  selector:
    matchLabels:
      istio: pilot
  namespaceSelector:
    any: true
  endpoints:
    - port: http-monitoring
      interval: 15s
```

### 5.2 分布式追踪

Istio 自动生成服务间调用的 Span，与业务 Span 联动：

```yaml
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: mesh-default
spec:
  tracing:
    - providers:
        - name: otel-tracing
      randomSamplingPercentage: 10  # 与业务采样率一致
```

### 5.3 访问日志

```yaml
apiVersion: telemetry.istio.io/v1alpha1
kind: Telemetry
metadata:
  name: mesh-default
spec:
  accessLogging:
    - providers:
        - name: envoy
      filter:
        expression: response.code >= 400  # 只记录非 2xx 日志
```

## 6. Istio 组件规模

### 6.1 资源需求估算

| 组件 | 副本 | 每副本资源 | 总计 |
|------|------|-----------|------|
| istiod | 2 | 1 CPU + 1GB | 2 CPU + 2GB |
| Ingress Gateway | 2 | 2 CPU + 2GB | 4 CPU + 4GB |
| Egress Gateway（可选） | 2 | 1 CPU + 1GB | 2 CPU + 2GB |
| Sidecar（每个业务 Pod） | — | 0.1 CPU + 128MB | 10 × 0.1 = 1 CPU + 1.28GB |

总计：约 **9 CPU + 10GB 内存**（10 个业务 Pod 时）

### 6.2 与 MaaS 现有基础设施的整合

| 设施 | Istio 集成方式 |
|------|--------------|
| Prometheus | 自动发现 `istio-proxy:15090` 指标 |
| Grafana | 导入 Istio 官方 Dashboard（7639/7636） |
| Jaeger | Istio 自动上报 Trace，无需额外配置 |
| Loki | Istio 访问日志通过 Fluentd/Promtail 采集 |
| ArgoCD | Istio 配置（VirtualService/DestinationRule）也可 GitOps 管理 |
| K8s HPA | 建议基于 Istio 指标（istio_requests_total）扩缩 |

## 7. 与 MaaS 各服务的对应关系

| MaaS 服务 | 相关 Istio 配置 |
|-----------|---------------|
| **gateway-service** | Ingress Gateway 替代 / 前置；中间件链中的限流、认证可与 Envoy ExtAuthz 集成 |
| **routing-service** | VirtualService 灰度路由；熔断配置 |
| **adapter-service** | 排除 Sidecar 注入，出站流量不通过 Mesh |
| **billing-service** | Kafka 流量不通过 Mesh（避免 Envoy 对 Kafka 协议兼容性问题） |
| **auth-service** | ExtAuthz 可下放到 Envoy 减少服务调用 |
| 其他服务 | 标准 Sidecar 注入，使用默认流量管理配置 |
