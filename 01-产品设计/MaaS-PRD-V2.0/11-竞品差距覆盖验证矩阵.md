## 第1章 验证说明

### 1.1 本文档的目的

本文档是 MaaS 平台 PRD V2.0 的第 11 号子文档，定位是**竞品差距的闭环验证记录**。  
核心目标：将 `竞品分析/28-25家竞品能力全景总结与MaaS差距分析.md`（以下简称"竞品分析报告"）中识别出的全部差距，逐项对应 V2.0 已发布子文档的章节，确认覆盖情况，并显式标注仍需跟进的领域，防止能力缺口在评审过程中被遗漏或隐性遗忘。

本文档服务以下受众和使用场景：
- **产品经理**：在 Sprint 计划时快速定位哪些竞品差距对应哪个子文档的哪个章节，避免重复定义或遗漏细节。
- **研发评审**：在架构评审或需求评审时，作为需求完整性的快速核查索引。
- **测试团队**：测试计划制订时，以本矩阵为输入，确保覆盖差距验收口径。
- **销售与解决方案**：与客户或合作伙伴讨论竞品对比时，作为能力叙事的支撑材料。

### 1.2 验证方法

本文档采用**逐项对照验证**方法，具体步骤如下：

**Step 1 差距识别**：从竞品分析报告 §17（PRD弱覆盖能力）、§18（产品路线建议）、§21（附：P0/P1/P2建议需求清单）三个章节提取原始差距条目，共识别 15 个主要差距域与 24 个 P0/P1/P2 建议。

**Step 2 映射确认**：逐差距确认在 V2.0 的 01-10 号子文档中是否存在对应章节，确认方式为章节标题可溯源、存在数据模型或状态机或验收口径中的至少一项。

**Step 3 覆盖状态判定**：按覆盖深度给出四级状态。

**Step 4 风险登记**：对部分覆盖或未覆盖条目，给出风险说明和建议行动。

### 1.3 覆盖状态标记说明

| 状态标记 | 含义 | 判定标准 |
|---|---|---|
| ✅ 全覆盖 | V2.0子文档存在对应章节且内容完整 | 有数据模型 + 状态机/流程图 + 验收口径三项中至少两项 |
| ⚠️ 部分覆盖 | V2.0子文档存在对应章节但内容不完整 | 章节存在，但数据模型或验收口径缺失，或仅为框架性描述 |
| ❌ 未覆盖 | V2.0子文档中无对应章节 | 无章节标题，无相关内容 |
| 🔄 计划中 | 已知需要覆盖，已纳入后续版本规划 | 在 00-总纲 或本文档中明确标注，待后续子文档承接 |

## 第5章 仍需关注的风险

### 5.1 高优先级待跟进项（需在V2.0评审阶段明确）

**风险1：开发者生态覆盖空白**  
当前V2.0缺少独立的开发者生态子文档，CLI/SDK/Terraform Provider/本地调试代理均在总纲中提及但无规格。在与 LiteLLM、Portkey 等竞品的开发者心智争夺中，开发者工具链是决定技术团队选型的关键因素。**建议在V2.1版本第一个 Sprint 启动 `12-开发者生态与SDK规格.md` 的编写**，优先覆盖 OpenAPI 3.1 规范发布、Python/Node.js SDK 和 CLI 基础命令集。

**风险2：Agent Trace规格不完整**  
04号LLMOps子文档覆盖了工具调用字段，但未定义 Agent 多步骤任务的 Trace 拼接视图（多轮 Tool Call → RAG → Memory → 最终响应的完整链路可视化）。随着客户 Agent 应用落地加速，这一能力缺失将影响 MaaS 平台在 Agent 场景的 LLMOps 竞争力。**建议在04号子文档 V1.1 补丁版本中新增 §Agent Trace 视图规格**。

**风险3：公开状态页规格缺失**  
03号子文档定义了 SLA 内部承诺和赔付口径，但缺少面向客户的公开状态页规格（类似 status.openai.com 的对外可见服务状态展示）。大客户在采购 SaaS 时通常要求提供公开状态页链接。**建议在09号Console子文档或运维手册中补充公开状态页规格**。

**风险4：合规报告格式的时效性**  
07号合规子文档定义了等保2.0/SOC2/ISO27001/GDPR报告模板，但合规标准本身会随监管变化而更新（如国内数据安全法实施细则、AI算法备案要求等）。**建议在合规子文档中建立报告模板版本化机制和年度复核流程**，避免合规报告内容滞后于实际监管要求。

**风险5：供应商市场的前期准备**  
P2建议中的供应商市场规划为V3.0，但供应商入驻协议、资质审核标准、测试准入环境等基础设施需要提前在V2.0阶段建立数据模型和管理规范。**建议在02号模型目录子文档中预留供应商入驻申请的数据字段设计**，为V3.0供应商市场能力打好基础，避免未来因数据模型不兼容而带来大规模重构。
