# MaaS 平台用户体验地图

**文档版本：** V1.0  
**最后更新：** 2026-05-20  
**文档归属：** 产品设计  
**覆盖角色：** 开发者用户 / 企业管理员

---

## 一、概述

### 1.1 文档目的

本文档通过用户体验地图（User Journey Map）梳理 MaaS 平台核心用户在完整生命周期中的行为、情绪与痛点，识别关键机会点，为产品优化和运营策略提供依据。

### 1.2 用户画像

#### 画像一：开发者（David）

| 属性 | 描述 |
|------|------|
| 角色 | 独立开发者 / 小团队后端工程师 |
| 技术背景 | 熟悉 Python/Node.js，有 REST API 调用经验 |
| 核心诉求 | 快速集成多家 LLM，控制成本，统一管理 API Key |
| 主要痛点 | 不同厂商 API 格式不统一，账单分散，调试困难 |
| 典型任务 | 接入 GPT-4o + DeepSeek，对比效果后切换，监控 Token 用量 |

#### 画像二：企业管理员（Alice）

| 属性 | 描述 |
|------|------|
| 角色 | AI 产品负责人 / IT 管理员 |
| 技术背景 | 不写代码，熟悉业务，关注合规与成本 |
| 核心诉求 | 管控团队用量，审批预算，保障数据安全 |
| 主要痛点 | 无法掌握各团队 LLM 使用情况，成本不可预测 |
| 典型任务 | 设置部门预算上限，审批 API Key 申请，查看月度账单 |

## 三、用户体验地图：开发者（David）

### 3.1 体验全景

| 维度 | 认知阶段 | 评估考虑 | 注册接入 | 活跃使用 | 深度依赖 | 推荐传播 |
|------|---------|---------|---------|---------|---------|---------|
| **时间跨度** | 第1天 | 第1-3天 | 第3-7天 | 第1-4周 | 第2-6月 | 第3月+ |
| **主要渠道** | 技术社区/搜索 | 官网/文档 | 注册页/控制台 | 控制台/API | 控制台/监控 | 社区/口碑 |

---

### 3.2 认知阶段

**用户行为**
- 在 GitHub、掘金、V2EX 浏览 LLM 相关讨论
- 搜索"多模型聚合 API"、"One API 替代方案"、"国内 GPT 代理"
- 看到 MaaS 平台的技术博客或广告
- 浏览 Twitter/X 上 KOL 分享的工具推荐

**用户想法**
> "每家 API 格式不一样，切换太麻烦了，有没有统一的方案？"  
> "接了好几个厂商的 Key，管理起来很乱，账单也分散。"  
> "这个 MaaS 平台是什么？看起来和 One-API 有点像，但功能更多？"

**情绪状态：** 困惑期待 ◑

**痛点问题**
- P1：信息触达不精准，开发者不容易在噪音中发现平台
- P2：与竞品差异不够清晰，第一眼看不出核心价值

**机会点**
- O1：在技术社区（掘金、GitHub Awesome 列表、V2EX）发布高质量技术文章
- O2：首页 Hero 区用具体数字（"统一接入 50+ 家模型，节省 60% 接入时间"）替代抽象描述
- O3：提供 One-API 迁移指南，降低认知门槛

---

### 3.3 评估考虑

**用户行为**
- 浏览官网产品介绍、功能列表、定价页
- 查看文档质量（Quick Start 是否够简单）
- 对比 One-API、LiteLLM 等竞品功能
- 在控制台注册前先看有无免费 Tier
- 查找 GitHub 仓库、Star 数量、Issues 活跃度

**用户想法**
> "文档写得清不清楚？能不能在 5 分钟内跑通第一个请求？"  
> "有没有免费额度试试？我不想一上来就充值。"  
> "安全吗？我的 API Key 会不会被泄露？"  
> "和 LiteLLM 比有什么优势？"

**情绪状态：** 审慎怀疑 ◔

**痛点问题**
- P1：文档层级混乱，Quick Start 不够突出
- P2：定价页信息复杂，用户无法快速判断自己的预计花费
- P3：安全机制说明不充分，开发者对 Key 托管有顾虑
- P4：与竞品对比缺乏客观的功能矩阵

**机会点**
- O1：文档首页做一个"5 分钟快速开始"卡片，突出最短路径
- O2：定价页增加"费用估算器"（输入每月 Token 量，自动计算费用）
- O3：安全页面专门说明 Key 存储方案（加密存储、不落日志）
- O4：增加 One-API / LiteLLM 对比页，直接展示差异

---

### 3.4 注册接入

**用户行为**
- 注册账号（邮箱/GitHub OAuth）
- 阅读 Quick Start 文档
- 在控制台创建第一个 API Key
- 配置第一个供应商（OpenAI / DeepSeek）
- 运行 SDK 示例代码，发送第一个请求
- 出错后查看错误信息并调试

**用户想法**
> "注册倒是挺快的，但在哪里加我的 OpenAI Key？找了一会儿。"  
> "第一个请求成功了！速度还可以。"  
> "报错了，错误信息说'Provider not configured'，但我明明配置了？"  
> "免费额度还剩多少？我怎么看？"

**情绪状态：** 先高后低 ◕ → ◔

```
情绪：  ▲
        |    ★ 首次请求成功
        |   /\
        |  /  \
        | /    \____  ← 遇到错误，信心受挫
        |/
        +──────────────► 时间
       注册  配置  发请求 调试
```

**痛点问题**
- P1：供应商配置路径不直观，新用户需要多次跳转才能找到
- P2：错误信息不友好，只说"配置错误"，不说哪里错了
- P3：免费额度余量不在显眼位置展示
- P4：SDK 示例代码需要自己填多处变量，容易出错

**机会点**
- O1：设计"注册即引导"流程，注册完自动跳转 Onboarding Checklist
- O2：错误信息增加"修复建议"（"可能原因：供应商 Key 未配置，点此前往配置 →"）
- O3：控制台首页放"今日可用额度"卡片，醒目显示
- O4：SDK 示例代码支持"一键复制已填充"（自动带入用户的 API Key）

---

### 3.5 活跃使用

**用户行为**
- 日常调用 API 处理业务需求
- 对比不同模型的输出质量（Playground）
- 在 Prompt 管理中心保存和迭代 Prompt
- 查看调用日志排查异常
- 在模型广场浏览新上线的模型
- 根据成本控制需求配置路由策略

**用户想法**
> "路由策略很好用，可以在 GPT-4o 贵的时候自动降级到 DeepSeek。"  
> "Playground 里对比模型效果太方便了，不用自己写脚本了。"  
> "日志查询功能缺少过滤条件，每次要翻很多才能找到我的请求。"  
> "想看某个模型的调用成功率，但监控图表不够细。"

**情绪状态：** 总体满意，偶有摩擦 ◕

**痛点问题**
- P1：日志查询功能过滤维度少，大量调用时查找效率低
- P2：路由策略配置界面不够直观，需要理解优先级逻辑
- P3：模型效果评估全靠主观，缺乏量化对比工具
- P4：Prompt 版本管理不完善，无法轻松回滚

**机会点**
- O1：日志增加"按应用/Key/模型/状态"多维过滤 + 全文搜索
- O2：路由策略增加"图形化预览"，显示流量分配示意图
- O3：Playground 增加多模型并排对比模式（同 Prompt 发多个模型）
- O4：Prompt 管理增加版本历史，支持 Diff 和一键回滚

---

### 3.6 深度依赖

**用户行为**
- 业务接入多个 AI 功能，依赖度高
- 开始关注成本优化，查看账单趋势
- 配置多层路由和预算告警
- 参与官方技术社区，提交 Feature Request
- 评估是否适合团队（从个人扩展到团队使用）

**用户想法**
> "这个平台已经是我 AI 项目的基础设施了，很难切换了。"  
> "希望能有更细粒度的成本归因，哪个业务线花了多少。"  
> "团队成员也想用，能不能加多人协作？"  
> "想看 SLA，上个月有几次可用性问题，影响了我的业务。"

**情绪状态：** 高度满意但有升级诉求 ◕◕

**痛点问题**
- P1：成本归因只到 Key 级别，无法按业务线/项目拆分
- P2：多人协作功能不完善，团队共享场景体验差
- P3：SLA 报告不透明，可用性事件通知不够及时
- P4：缺乏 Enterprise 级别的功能（SSO、审计日志、私有化）

**机会点**
- O1：引入"项目"概念，支持按项目维度统计费用
- O2：推出团队/组织功能，支持角色权限管理
- O3：增加 StatusPage（可订阅），主动推送可用性事件通知
- O4：规划企业版 + 私有化部署路线图，主动联系高用量用户

---

### 3.7 推荐传播

**用户行为**
- 在团队内部推荐，帮助同事完成配置
- 在技术社区分享使用心得（博客/短视频）
- 向 GitHub 仓库提交 Star
- 在社交媒体引用 MaaS 平台的内容

**用户想法**
> "我在博客写了一篇接入教程，顺便推荐了这个平台。"  
> "同事用了一段时间后也觉得好用，决定付费升级。"  
> "要是有邀请奖励就更好了，可以帮我抵点费用。"

**情绪状态：** 积极正向 ◕◕

**痛点问题**
- P1：没有推荐计划，用户自发推荐缺乏激励
- P2：分享的素材（截图/数据）不够"好看"，不利于传播

**机会点**
- O1：推出"邀请好友返佣"计划（被邀请者首充返 X%）
- O2：账单/统计页增加"生成分享卡片"功能，用数据说话
- O3：官方整理"用户案例"合集，联系高活跃用户做访谈

## 四、用户体验地图：企业管理员（Alice）

### 4.1 体验全景

| 维度 | 认知阶段 | 评估考虑 | 接入部署 | 管控运营 | 深度治理 | 合规续约 |
|------|---------|---------|---------|---------|---------|---------|
| **时间跨度** | 第1周 | 第1-2周 | 第2-4周 | 第1-3月 | 第3-12月 | 年度 |
| **决策者** | IT/AI 负责人 | 采购+IT | IT+安全 | 管理员 | 管理员+财务 | CTO+财务 |

---

### 4.2 认知阶段

**用户行为**
- 收到开发团队反馈，需要统一管控 LLM 使用
- 调研 MaaS 平台、API 网关等方案
- 观看产品演示视频或参加线上发布会

**用户想法**
> "开发团队各自接 API，我完全不知道花了多少钱，用了什么数据。"  
> "有没有一个平台可以统一管控，还能生成报表给领导看？"

**情绪状态：** 需求迫切，主动寻解 ◕

**痛点问题**
- P1：产品介绍太偏技术，管理员看不到"合规管控"的价值

**机会点**
- O1：面向管理员的 Landing Page，重点突出：合规审计、成本管控、权限管理
- O2：提供企业案例（某企业接入后减少 X% AI 成本）

---

### 4.3 接入部署

**用户行为**
- 组织 POC 测试，IT 工程师配置，管理员验收
- 对接企业 SSO（LDAP/SAML）
- 设置组织架构（部门/项目组/成员权限）
- 配置合规策略（数据不出境、内容安全）

**用户想法**
> "SSO 对接花了两天，文档不够完整，要靠技术支持才搞定。"  
> "组织架构导入支持批量导入吗？手动一个个加太慢了。"

**情绪状态：** 有挫折感 ◔

**痛点问题**
- P1：SSO 配置文档不完整，缺少主流 IdP（企业微信/Okta）的详细配置步骤
- P2：组织架构无法批量导入（Excel/AD 同步）

**机会点**
- O1：企业微信/Okta/飞书 SSO 对接文档专项完善
- O2：支持 CSV 批量导入用户，支持 LDAP/AD 同步

---

### 4.4 管控运营

**用户行为**
- 日常审批开发者的 Key 申请
- 查看各部门的 Token 消耗报表
- 处理内容安全告警（敏感词触发）
- 设置月度预算告警

**用户想法**
> "审批请求太多了，每天要处理十几条，能不能设置自动审批规则？"  
> "部门报表不够灵活，导出的 Excel 还要自己再加工。"

**情绪状态：** 稳定但效率偏低 ◑

**痛点问题**
- P1：审批流程无自动化规则，全靠人工
- P2：报表导出格式固定，财务部门需要的字段不够灵活
- P3：内容安全告警量大时，难以区分优先级

**机会点**
- O1：审批中心支持"自动批准规则"（低于阈值金额自动批准）
- O2：报表支持自定义字段和多格式导出（Excel/CSV/PDF）
- O3：内容安全告警支持"严重/中/低"分级，支持批量处理

---

### 4.5 深度治理

**用户行为**
- 出具年度 AI 使用合规报告
- 根据供应商价格变化调整路由策略
- 评估私有化部署可行性

**情绪状态：** 主动管理，期待更强工具 ◕

**机会点**
- O1：一键生成合规报告（包含数据流向、用量、安全事件摘要）
- O2：私有化部署专项方案页面，降低决策门槛
