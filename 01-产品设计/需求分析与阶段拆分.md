# MaaS平台需求分析与阶段拆分

**基于文档：** 产品设计/maas平台PRD文档.md  
**输出目的：** 将PRD转为可执行的版本规划、需求明细与交付清单。  
**当前日期：** 2026-05-15

---

## 1. 需求分析结论（Executive Summary）

### 1.1 核心业务目标
1. 建立统一OpenAI兼容入口，降低多模型接入复杂度。
2. 通过智能路由与缓存实现性能与成本优化。
3. 打通计费、监控、权限、审计，形成企业级运营闭环。
4. 支撑从内测到GA的可控发布与商业化验证。

### 1.2 需求分层
1. **P0 基础能力层**：网关、协议适配、模型管理、路由、基础计费与监控。
2. **P1 治理增强层**：预算治理、Policy as Code、Prompt实验、质量-成本联动分析。
3. **P2 规模化能力层**：容灾演练、自动化评测、私有化一体机等扩展能力。

### 1.3 非目标（本期不做）
1. 大规模分布式预训练平台。
2. 边缘联邦学习能力。
3. AI应用商店运营体系。

## 2.2 各阶段详细拆分

### Phase 0 预备期（联调与门禁）
**目标：** 降低后续集成风险，建立统一工程基线。  
**范围：**
1. API规范冻结（OpenAI兼容字段、错误码、限流策略）。
2. 数据字典落盘（请求、计费、审计、监控字段）。
3. 观测与日志规范（request_id贯通、关键埋点事件）。
4. 环境与发布规范（开发/测试/预发/生产）。

**交付件：**
1. API契约文档（可生成SDK）。
2. 数据字典与事件字典。
3. 冒烟测试清单与回归测试计划。
4. 发布回滚SOP。

**退出条件：**
1. 关键接口联调通过率100%。
2. 日志与指标链路可查询。
3. P0问题清零。

### Phase 1 MVP内测（P0闭环）
**目标：** 完成MVP最小闭环并进行封闭内测。  
**范围：**
1. 统一API网关与协议适配器。
2. 模型广场、模型接入与部署基础能力。
3. 路由引擎、故障转移、基础语义缓存。
4. 计量计费、监控告警、基础RBAC。

**交付件：**
1. 控制台基础版（开发者端+管理端）。
2. SDK与快速入门文档。
3. 内测报告（稳定性、性能、成本、反馈）。

**退出条件：**
1. 内测核心API成功率>=99.5%。
2. 无P0/P1阻塞缺陷。
3. 计费准确性可对账。

### Phase 2 Beta公开（稳定与增长验证）
**目标：** 面向公开开发者验证可用性与增长指标。  
**范围：**
1. 高并发稳定性优化与容量扩展。
2. 路由策略效果优化（命中率、延迟、成本）。
3. 运营看板与告警规则完善。
4. 开发者文档、FAQ、支持流程完善。

**交付件：**
1. 公开Beta发布说明。
2. 运营周报模板与问题处理机制。
3. 用户反馈闭环系统。

**退出条件：**
1. 可用性>=99.9%。
2. P95延迟<=1秒。
3. 周活跃留存>60%。

### Phase 3 GA发布（企业可交付）
**目标：** 达到企业生产可用与商业签约能力。  
**范围：**
1. 审计与权限治理增强。
2. SLA监控与违约预警机制。
3. 客户迁移方案与交付手册。
4. 市场宣导与标杆客户方案。

**交付件：**
1. GA版本说明与SLA协议。
2. 企业客户上线手册与运维手册。
3. 商业化报表（MRR、转化、留存）。

**退出条件：**
1. 首批企业签约>=10家。
2. SLA达标，重大事故可控。
3. 客户成功案例可复用。

### Phase 4 增强迭代（竞品驱动）
**目标：** 对齐Portkey/Helicone等能力，强化差异化。  
**范围：**
1. Prompt实验与回归中心（A/B、离线评测）。
2. Policy as Code中心（版本化、审批、灰度、回滚）。
3. 质量-成本联合分析看板。
4. 多区域容灾与季度演练。

**退出条件：**
1. 每项增强能力有上线报告。
2. 实际指标改善可量化。
3. 客户复购/续约有正向提升。

## 5. 关键依赖与风险

### 5.1 外部依赖
1. 模型供应商稳定性与商务条款。
2. SSO与权限中心对接资源。
3. 财务系统账单接口对接周期。
4. GPU资源与多区域基础设施可用性。

### 5.2 主要风险与缓解
1. **路由效果不达标**：先上线保守策略，再逐步放量A/B。
2. **计费争议**：先建立对账机制，再开通企业账单。
3. **上线波动**：灰度发布+自动回滚+战情室机制。
4. **需求蔓延**：严格执行阶段退出条件与变更审批。

## 8. 逐项需求详细说明（可直接拆任务）

说明：本节按“目标-场景-流程-输入输出-异常-验收”展开，可直接用于PRD评审和研发/测试分工。

## 8.1 ACC-001 统一API网关（P0）
**需求目标：** 对外提供统一OpenAI兼容入口，屏蔽多厂商差异。  
**用户场景：** 开发者使用现有OpenAI SDK，仅替换`base_url`和`api_key`后完成迁移。  
**主流程：**
1. 接收请求并校验鉴权头。
2. 执行限流与基础参数校验。
3. 生成`request_id`并写入链路上下文。
4. 转发到路由引擎。

**输入：** `model/messages/temperature/max_tokens/stream`等标准字段。  
**输出：** 标准化响应或标准错误结构（401/403/429/5xx）。  
**边界与规则：**
1. 不支持字段必须返回明确错误码和错误文案。
2. 流式响应必须支持断线可追踪。

**异常处理：**
1. 鉴权失败 -> 401。
2. 超频 -> 429并附限流说明。
3. 上游不可用 -> 5xx并记录故障原因。

**非功能要求：**
1. 单实例QPS满足基线压测目标。
2. 日志写入延迟不影响主请求。

**验收用例（最少）：**
1. 用Python/Node SDK调用成功。
2. 无效Key返回401且不转发上游。
3. 限流命中返回429，含可读提示。

## 8.2 ACC-002 协议适配器（P0）
**需求目标：** 将统一协议映射到不同模型供应商参数。  
**用户场景：** 同一请求可路由到OpenAI/Anthropic/国内模型。  
**主流程：**
1. 根据目标供应商选择适配器。
2. 执行参数映射和字段校验。
3. 执行错误码反向映射。

**输入：** 统一请求体 + 目标供应商标识。  
**输出：** 供应商请求体 + 平台统一响应体。  
**边界与规则：**
1. 映射规则需版本化管理。
2. 新供应商接入必须配套映射测试集。

**异常处理：** 映射失败返回`adapter_mapping_error`并附字段原因。  
**非功能要求：** 映射处理耗时占比不超过总时延5%。  
**验收用例：**
1. 多供应商同语义请求结果结构一致。
2. 错误码映射覆盖率>=95%。

## 8.3 ACC-003 SDK自动生成（P1）
**需求目标：** 降低开发者接入成本，减少手写客户端错误。  
**主流程：** OpenAPI发布 -> CI生成SDK -> 发布包管理器 -> 文档同步。  
**输入：** 冻结后的OpenAPI规范。  
**输出：** Python/Node/Go SDK包及示例。  
**验收用例：**
1. 三种语言示例均可跑通。
2. 版本号和API变更日志一致。

## 8.4 MOD-001 模型广场（P0）
**需求目标：** 支持模型发现、对比和快速试用。  
**用户场景：** 开发者按价格、延迟、能力标签筛选模型。  
**主流程：**
1. 拉取模型元数据。
2. 组合筛选条件并排序。
3. 展示对比与试用入口。

**输入：** 模型元数据、性能指标、价格信息。  
**输出：** 模型列表/详情/对比结果。  
**验收用例：**
1. 筛选结果与条件一致。
2. 试用调用链路可追踪到账单与日志。

## 8.5 MOD-002 精调工作台（P0）
**需求目标：** 提供从数据上传到训练观察的一站式精调入口。  
**主流程：**
1. 数据集上传与格式校验。
2. 训练参数配置（LoRA/QLoRA）。
3. 任务启动与状态追踪。
4. 训练产物归档。

**输入：** 数据集文件、训练参数、目标基础模型。  
**输出：** 训练任务记录、指标曲线、模型产物版本。  
**异常处理：** 数据格式错误、资源不足、训练中断。  
**验收用例：**
1. 支持任务失败重试。
2. 训练日志可定位错误行。

## 8.6 MOD-003 模型部署上线（P0）
**需求目标：** 支持模型版本发布、灰度与回滚。  
**主流程：**
1. 选择模型版本。
2. 配置灰度比例与目标流量。
3. 监控关键指标。
4. 自动/手动回滚。

**输入：** 模型版本、灰度策略、阈值规则。  
**输出：** 发布状态、指标看板、回滚记录。  
**验收用例：**
1. 灰度比例按配置生效。
2. 阈值超限可自动回滚。

## 8.7 ROU-001 智能路由引擎（P0）
**需求目标：** 在可用性约束下选择综合最优模型。  
**主流程：**
1. 读取策略与候选模型集。
2. 计算成本/时延/稳定性得分。
3. 输出最终路由结果与原因。

**输入：** 请求特征、模型实时指标、策略参数。  
**输出：** 目标模型、决策原因、候选打分明细。  
**边界与规则：**
1. SLA优先于成本。
2. 降级策略必须可配置。

**验收用例：**
1. 在高负载下路由命中率稳定。
2. 每次决策可审计可回放。

## 8.8 ROU-002 故障转移（P1）
**需求目标：** 上游故障时自动切换，降低失败率。  
**主流程：** 失败检测 -> 候选切换 -> 重试 -> 结果返回。  
**触发条件：** 超时、429、5xx、健康检查失败。  
**输出：** 切换结果、重试次数、最终状态。  
**验收用例：**
1. 故障场景能自动切换成功。
2. 切换过程对调用方透明。

## 8.9 ROU-003 语义缓存（P2）
**需求目标：** 通过缓存命中降低Token成本和延迟。  
**主流程：** 一级哈希匹配 -> 二级向量匹配 -> 回源写缓存。  
**输入：** 规范化Prompt、embedding向量、阈值配置。  
**输出：** 缓存命中结果、节省成本估算。  
**关键规则：**
1. 命中阈值可按场景配置。
2. 支持TTL/LRU和手动清理。

**验收用例：**
1. 命中率、节省Token可量化。
2. 误命中率在可接受阈值内。

## 8.10 OPS-001 计量计费系统（P1）
**需求目标：** 提供可对账、可审计、可追溯的Token计费。  
**主流程：** 采集调用明细 -> 计算费用 -> 生成账单 -> 对账输出。  
**输入：** prompt/completion token、模型单价、客户维度。  
**输出：** 日账单、月账单、对账差异报告。  
**异常处理：** 计费服务不可用时进入补录队列。  
**验收用例：**
1. 账单可导出且字段完整。
2. 对账差异在容忍阈值内或有补偿机制。

## 8.11 OPS-002 监控告警中心（P1）
**需求目标：** 实现性能、稳定性、成本的统一告警。  
**主流程：** 指标采集 -> 规则计算 -> 告警触发 -> 通知与归档。  
**输出：** 告警事件、处理状态、MTTR统计。  
**验收用例：**
1. 关键告警可在阈值内触发。
2. 告警可追踪到处理人和处理时长。

## 8.12 OPS-003 权限与审计（P1）
**需求目标：** 保证多租户隔离和关键操作可审计。  
**主流程：** 角色授权 -> 权限校验 -> 操作记录 -> 审计查询。  
**输入：** 用户、角色、资源、动作。  
**输出：** 许可结果、审计日志。  
**验收用例：**
1. 越权访问被拦截。
2. 审计日志满足时间、用户、资源、动作四要素检索。

## 8.13 EXP-001 Prompt实验中心（P1）
**需求目标：** 提升提示词迭代效率与质量稳定性。  
**主流程：** 建立实验 -> 划分流量 -> 收集效果 -> 显著性分析 -> 决策发布。  
**输入：** Prompt版本、评测集、实验流量。  
**输出：** 实验报告、推荐版本。  
**验收用例：**
1. 支持A/B实验结果对比。
2. 支持回归评测并生成报告。

## 8.14 GOV-001 Policy as Code中心（P1）
**需求目标：** 将策略管理从人工配置升级为可治理流程。  
**主流程：** 提交策略变更 -> 审批 -> 灰度 -> 全量 -> 回滚。  
**输入：** 策略代码/配置、审批人、发布范围。  
**输出：** 策略版本、发布记录、审计轨迹。  
**验收用例：**
1. 策略变更需审批才能生效。
2. 支持按项目/API Key范围灰度。

## 8.15 GOV-002 预算审批工作流（P1）
**需求目标：** 把预算预警升级为可执行的费用控制。  
**主流程：** 预算设定 -> 超限拦截 -> 提交审批 -> 批准放行/拒绝。  
**输入：** 预算阈值、项目负责人、审批规则。  
**输出：** 审批单、拦截日志、放行记录。  
**验收用例：**
1. 超预算请求默认拦截。
2. 审批通过后限定时间内放行。

## 8.16 OBS-001 质量-成本联合看板（P1）
**需求目标：** 同时评估模型效果与成本，支撑运营决策。  
**主流程：** 指标汇聚 -> 维度建模 -> 看板渲染 -> 下钻分析 -> 周报导出。  
**输入：** 质量指标、时延、Token成本、租户维度。  
**输出：** 多维图表、异常归因、导出报表。  
**验收用例：**
1. 支持按模型/项目/租户下钻。
2. 周报导出内容与看板一致。

## 8.17 REL-001 多区域容灾演练（P2）
**需求目标：** 提升平台连续性与企业信任。  
**主流程：** 演练计划 -> 脚本切换 -> 业务验证 -> 回切 -> 复盘报告。  
**输入：** 演练窗口、切换脚本、验证清单。  
**输出：** RTO/RPO结果、问题清单、改进项。  
**验收用例：**
1. 每季度完成一次演练。
2. RTO/RPO达到预设目标并有证据链。

## 9. 研发拆解建议（每个需求的标准子任务）

每个需求ID建议固定拆成以下子任务，避免漏项：
1. 产品：流程图、字段定义、边界说明、验收标准。
2. 后端：接口实现、错误码、日志与审计、性能优化。
3. 前端：页面交互、状态处理、错误提示、埋点上报。
4. 测试：功能、异常、回归、性能与安全测试。
5. 运维：监控告警、发布回滚、容量与SLA校验。
6. 文档：API文档、使用手册、变更记录。
